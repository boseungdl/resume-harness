이 스킬은 이 저장소의 **자소서 하네스**의 오케스트레이터다. 리서치 → 모범답안 → 개인화의 3단계를 몰고, **앞 두 단계는 서브에이전트(Task 도구)에 위임**한다. **개인화·작성(③)은 메인 에이전트가 직접** 한다 — 프로필은 크므로 메인에서 한 번만 로드하고, 무엇보다 **여러 문항이 하나의 인생 축을 공유(S1)하려면 한 컨텍스트에서 통으로 써야** 하기 때문이다. (문항별로 서브에이전트를 나눠 쓰면 축이 깨져 문항이 따로 논다 — 서브에이전트는 쓰기가 아니라 검사에 쓴다.)

**규칙은 2계층이다.** ③ 생성에는 **서사 규칙(S1~S6) + 문항별 카드**를 적용한다(서사는 사후에 못 넣으므로 생성 전략). **폴리시(문체 금지 항목)와 최종 검증은 `review-jaso`**가 맡는다 — 원문은 거기 한 곳에만 둔다(중복 방지). 생성에 유지되는 근거 제약은 **자유도**(사실 안전장치)와 자수다. (설계: ADR-0006·0007)

## 시작 전 로드
- **용어집**: `CONTEXT.md`.
- **프로필**: `docs/profile/about-me.md` — 지원자의 유일한 마스터 소스. ③에서 메인이 한 번 로드. §3(직무별 강조)·§4~6(성장·여정·장면)을 서사 재료로.

## 자유도 (유일하게 생성에 적용되는 근거 제약)
기본 **낮음(프로필 기반)** — 프로필 사실만 근거로, 없는 경험·수치는 만들지 않는다. 실제 지원은 낮음. '높음'은 연습용. 시작 시 확인. (ADR-0003)

## 생성 서사 규칙 (③에 전면 적용 · 메인)
자소서를 STAR 일화 나열이나 프로필 항목 재서술로 쓰지 않는다.

- **S1. 하나의 관통하는 인과 서사 (누적).** "왜 이 길·무엇을 원하나"를 축으로 각 단계가 다음의 **동기**가 되게. 한 사건으로 직무를 정했다 식 금지 — 넓게 배움 → 실무로 좁힘 → 프로젝트로 굳어짐. 문항이 여럿이어도 같은 인생 축을 공유.
- **S2. 사실 → 의미 번역.** 프로필 항목(자격증 목록·기술 사례·산출물)을 그대로 문장화하지 말고, 서사에서 **무엇을 증명하는지**로 바꾼다. 겪은 사람만 쓸 수 있는 **구체 장면 하나**로. (용어 나열·속빈강정 금지)
- **S3. 강점·성품은 선언 말고 상황·행동으로.** "구조적 사고가 강점입니다" ✗ → "이 일이 전체를 먼저 보는 능력을 실제로 요구했다." 본인이 자기 것으로 인지 못 하는 추상 성품·가치는 선언하지 말고 행동으로 증명하거나 삭제.
- **S4. 편력·약점은 여정으로, 약점 과노출 금지.** 정직하게 인정하되 강점 서사에 복무하게. **정직함이 목적이 아니라 강점 서사가 목적** — 사실을 담담히 나열해 약점만 돋보이면 실패(지원자 핵심 맥락은 review-jaso 참조).
- **S5. 자소서는 서사, 기술 상세는 이력서·포폴.** 스택·수치·해결 과정을 깊이 풀지 않고 서사 앵커로만. 기법 나열(격리·재시도·멱등·캐싱 스펙 문서화) 대신 **저울질한 트레이드오프 하나를 깊게.**
- **S6. 장면으로 열고 체현한다.** telling으로 열지 않는다 — "처음 알았다/재미있었다" 발견담 금지. 겪은 장면에서 시작해 "~이 제가 코드를 대하는 방식입니다" 톤으로. **결론은 먼저 서되 장면으로 체현**한다(첫 문장에 결론만 던지는 순수 두괄식은 쓰지 않는다).

## 문항별 카드 (③에서 해당 문항에 맞춰 적용)
각 문항은 고유의 목적이 있다. 아래 카드로 문항을 맞춘다. (표준 5문항 기준 · 실제 문항에 맞게 조정)

- **① 지원 동기** — 동기는 동기다(자기소개 아님). 끌림·하고 싶음의 언어로 **①왜 끌리나 → ②진짜 뿌리 → ③여기서 뭘 하고 싶나.** *금지*: 억지 origin(무관한 과거 엮기), 부정형 오프닝("저는 ~한 사람이 아닙니다"), 검색되는 기업정보 나열.
- **② 성장 과정** — 지금의 일하는 방식이 **어디서 왔는지**를 인과로. *금지*: 유년기 미담·연대기 나열.
- **③ 성격의 장단점** — 장·단점을 **같은 성향의 양면**으로, 행동 장면으로 증명. *금지*: 형용사 선언("꼼꼼합니다"), 단점을 위장한 자랑("완벽주의라").
- **④ 직무 역량·핵심 경험** — 한 경험의 **판단·트레이드오프**를 서사로(기술 상세는 포폴). *금지*: 기법 나열, 방어 못 할 스택 name-drop.
- **⑤ 입사 후 포부** — 현실적 성장 궤적(단기/중기)을 직무와 연결. *금지*: 막연한 다짐·거창한 비전.

> 위 ②~⑤ 카드는 초안이다 — 사용자 지침에 맞춰 계속 다듬는다. ①은 사용자가 확정한 형태.

## 입력 받기
1. **공고**: 본문/경로. 없으면 **범용 모드** — 지원 **직무**를 받는다.
2. **문항**: 문항 목록·자수 제한. 범용 모드에서 없으면 표준 5문항 제안(위 카드 순서).

## 작업 폴더
`.scratch/<회사-직무-slug>/` (범용 모드는 `<직무-slug>`): `posting.md` · `research.md`(①) · `exemplar.md`(② 서사 틀) · `자소서.md`(③ 최종).

## 파이프라인

### ① 리서치 (서브에이전트 ×1, 웹)
공고(또는 직무)를 넘겨 **웹으로 조사**: 기업(사업·인재상·최근 이슈), 직무 실제 업무, **합격 자소서 패턴**, JD. 요구역량·키워드·핵심 업무를 구조화, 지원 직무를 §3 직무군으로 **추론**해 제시. `research.md` 저장.
- **최신 우선**: **2025년 이후 자료만 참고** — 트렌드가 빨리 바뀌므로 그 이전은 배제. 게시일·연도 확인, 불명확하면 신뢰도 하향.
- 규칙·프로필 로드하지 않음 — 소스 수집만.
- **사용자 확인**: 추론 직무 승인.

### ② 모범답안 (서브에이전트 ×1)
`research.md`만 주고 문항별 **자유 모범답안**을 쓰게 한다 — 프로필 무관 창작. ③에서 갈아끼울 **틀**이며 핵심은 **서사 구조**(인과 흐름·설득 각도)다. 기술 상세는 최소화하고(S5) 기법 나열 대신 트레이드오프 하나를 깊게. `exemplar.md` 저장.

### ③ 개인화·작성 (메인, 통으로)
메인이 먼저 **인생 축(S1)을 잡고** 시작한다. `exemplar.md`의 서사 구조를 참고하되 내용은 **프로필 실제 경험**으로 채우고, **§생성 서사 규칙 S1~S6 + 문항별 카드**를 적용한다. 매칭 표는 만들지 않는다.
- **형식**: 합니다체. 오프닝·구조는 S6을 따른다(장면 체현, 순수 두괄식 아님).
- **자수**: 기본 공백 포함. "공백 제외" 명시 문항만 예외. 각 답변 끝에 실제 글자 수 표기(예: `[412자 / 공백 포함]`).
- 문항이 많아도 **한 컨텍스트에서 통으로** 써 인생 축을 공유. `자소서.md`에 문항 순서대로.

### ④ 점검 → review-jaso
자수 초과·문항 간 중복 등 기계적 점검만 한다. **서사 규칙 준수 + 폴리시(AI 문어체·비유·name-drop 등) 검증은 `review-jaso`**로 넘긴다 — 문항별 병렬 검사가 필요하면 review 단계에서 서브에이전트로 팬아웃한다(쓰기가 아닌 검사에만).

## 수정 루프
특정 문항 수정 → 그 문항만 ③ 재작성(단, 인생 축 유지). 틀을 바꾸려면 ② 재실행. 소스 부족하면 ①로.
