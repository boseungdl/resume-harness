이 스킬은 이 저장소의 **자소서 하네스**의 오케스트레이터다. 공고 분석 → 매칭 → 문항별 작성의 3단계를 몰고, 각 단계를 서브에이전트(Task 도구)에 위임하며, 사람이 검토하는 지점을 관리한다.

시작 전 배경 문서를 읽는다:
- **프로필**: `docs/profile/about-me.md` — 지원자의 유일한 마스터 소스. §3(직무별 강조 매핑)을 매칭 가이드로 쓴다.
- **용어집**: `CONTEXT.md` — 하네스·프로필·자소서·문항·자수·매칭 용어를 이 정의대로 쓴다.
- **결정 기록**: `docs/adr/` — 특히 0003(근거 정책).

## 핵심 원칙

- **근거 정책(ADR-0003)**: 프로필에 관련 내용이 있으면 적극 활용한다. 프로필 밖 내용의 창작도 제약하지 않는다(완전 자유). 사실 검증은 제출 전 본인 몫이다.
- **기본 문체**: 합니다체 · 두괄식(첫 문장에 핵심) · STAR 근거(상황-행동-결과, 추상 형용사보다 사실·수치) · 문항당 핵심 경험 1~2개 집중. 문항별로 덮어쓸 수 있다.
- **자수**: 기본은 **공백 포함**으로 센다. 문항이 "공백 제외"를 명시하면 그 문항만 규칙을 바꾼다. 각 문항 답변 끝에 **실제 글자 수를 표기**한다(예: `[412자 / 공백 포함]`).

## 입력 받기

1. **공고**: 사용자가 공고 본문을 붙여넣거나 파일 경로를 준다.
   - 공고가 없으면 **범용 모드**로 전환 — 사용자에게 지원 **직무**를 받는다. 이 직무의 일반 요구역량 분석이 공고를 대체한다.
2. **문항**: 사용자가 문항 목록과 자수 제한을 준다(예: "지원동기 500자, 성장과정 800자").
   - 범용 모드에서 문항이 없으면 표준 5문항을 기본값으로 제안한다: ① 지원 동기 ② 성장 과정 ③ 성격의 장단점 ④ 직무 역량·핵심 경험 ⑤ 입사 후 포부. 사용자가 편집할 수 있다.

## 작업 폴더

지원 건마다 폴더 하나: `.scratch/<회사-직무-slug>/` (범용 모드는 `<직무-slug>`).
산출물을 여기에 저장한다:
- `posting.md` — 공고 원문 (범용 모드는 생략, 직무를 여기 기록)
- `analysis.md` — ①단계 산출물
- `matching.md` — ②단계 산출물
- `자소서.md` — ③단계 최종 답변 (자수 표기 포함)

## 파이프라인

### ① 분석 (서브에이전트 ×1)
공고(또는 직무)를 서브에이전트에 넘겨 **요구역량 · 인재상/키워드 · 직무 핵심 업무**를 구조화해 뽑는다. 지원 직무를 `docs/profile/about-me.md` §3의 직무군 중 하나로 **추론**해 함께 제시한다. `analysis.md`에 저장.

**사용자 확인 지점**: 추론한 지원 직무를 사용자에게 제시하고 승인받는다(범용 모드는 사용자가 직무를 이미 선언). 상충 없이 진행.

### ② 매칭 (서브에이전트 ×1)
승인된 직무 기준으로, 분석에서 뽑은 요구역량 **각각**에 프로필의 어떤 경험·성과(프로젝트 번호·§ 참조)를 근거로 댈지 표로 매핑한다. §3 전면배치 자산을 우선 활용. `matching.md`에 저장.

**사용자 검토 지점**: 매칭 표를 사용자에게 보여주고 "이 경험 말고 저걸 써" 식의 수정을 받는다. **확정된 매칭만** 작성의 근거가 된다.

### ③ 작성 (서브에이전트 ×N — 문항당 하나, 병렬)
문항이 여러 개면 문항 하나당 서브에이전트 하나를 **병렬로** 띄운다. 각 에이전트는 확정된 매칭 + 프로필을 받아 해당 문항의 답변을 기본 문체로 작성하고, **자수 제한을 준수**하며 실제 글자 수를 세어 표기한다. 결과를 `자소서.md`에 문항 순서대로 모은다.

### ④ 검증
완성본 전체를 훑어 **자수 초과 · 문항 적합성 · 진부한 표현 · 문항 간 중복**을 점검하고, 문제가 있으면 해당 문항을 다시 돌린다. 최종본을 사용자에게 제시한다.

## 수정 루프
사용자가 특정 문항의 수정을 요청하면 그 문항만 ③을 다시 돌린다. 매칭 자체를 바꾸려면 ②로 돌아간다.
