## 🎯 30초 핵심 (누구나 이해되게)

**에이전트(AI)는 책상 하나로 일하는 사람과 같다.** 책상 위에 올려둔 것만 보고 답한다.

```
   ┌─────────────────────────────────────────┐
   │   책상 (컨텍스트 창) — 크기가 정해져 있다   │
   │                                           │
   │   [규칙]  [읽은 파일]  [대화]  [검색결과]   │  ← 여기 있는 것만 보고 답한다
   │                                           │
   └─────────────────────────────────────────┘
         ▲ 올린다                  │ 답한다
         │                        ▼
   파일·메모리·검색            사람에게 응답
```

여기서 나오는 결론은 딱 넷:

| # | 사실 | 그래서 |
|---|---|---|
| 1 | **책상은 하나·유한하다** | 아무거나 많이 올리면 정작 중요한 걸 놓친다 |
| 2 | **무관한 게 섞이면 성능이 떨어진다** | "많이"가 아니라 "필요한 것만 깨끗하게" |
| 3 | **책상이 꽉 차면 오래된 걸 요약해 치운다(손실)** | 중요한 건 **파일에 따로 적어둬야** 안 사라진다 |
| 4 | **일이 끝나 책상을 비우려면 통째로 치우는 수밖에 없다** | 작업이 바뀌면 **새 세션으로 끊는다** |

> **한 문장:** 책상을 깨끗하게 — 필요한 것만 올리고(예방), 무관해지면 세션을 끊어 내리고(리셋), 넘길 건 파일에 남긴다.

## 📊 한눈에 보기 (전체 도식)

```
      올림 ▲ (문턱 = 라우터)                     내림 ▼ (배수구 = 세션 리셋)
           │                                                │
파일·메모리·스킬·도구출력  ──▶   [ 창 : 유한한 상자 하나 ]   ──▶   응답
                                        │
              창을 채우는 3겹  ─────────┤
                ① 전문 고정  = 늘·무조건 지킬 규칙   → 항상 상주 (CLAUDE.md 앞)
                ② 라우터     = "이런 게 있다" 한 줄  → 한 줄만 상주
                ③ 본문       = 이번 작업 상세         → 필요할 때만 꺼냄

  ▪ 올라온 건 매 턴 비용(용량·주의) · 무관하면 노이즈 · 압축되면 손실(되돌리기 X)
  ▪ 지킬 규칙은 앞이나 끝에 (가운데는 묻힌다)
  ▪ 대량 원문은 서브에이전트(별도 창)에 던지고 → 요약만 회수
```

**핵심 5줄**
1. 창은 **하나·유한**하다 — 들어간 건 매 턴 다시 처리되고, 무관할수록 초점이 흐려진다.
2. 그래서 **3겹으로**: 늘 지킬 것만 전문 고정, 나머지는 라우터로 가리켜 **필요할 때만** 펼친다.
3. **자리도 실력**: 반드시 지킬 지시는 창의 앞이나 끝에 둔다(가운데는 놓치기 쉬움).
4. **세션이 유일한 배수구**: 작업이 바뀌면 끊고, 넘길 건 **파일에 고정**(압축은 손실적).
5. **격리는 "분해되는 일"에**: 단계가 깔끔히 나뉘어 파일로 떨어지면 서브에이전트에 위임하고 결과만 회수. 맥락을 계속 이어야 하는 핵심 작업은 메인에 두고 **주변부(탐색·대량 읽기)만** 정찰 위임.

> **배치 한 줄 규칙 — 자주+짧다→상주 / 가끔+크다→검색 / 항상+절대→전문 고정 / 대량 원문→격리.**

## Part 1. 동작 원리

에이전트를 이해하는 최소 모델. **정적(한 턴)** 과 **동적(여러 턴=루프)** 으로 나눈다.
핵심 주장은 Anthropic 공식 자료로 뒷받침한다(→ [출처](#출처)).

### 1A. 정적 — 한 턴 안에서

**원리 1 — 창은 하나이고 유한하다**
에이전트가 한 대화에서 볼 수 있는 건 **정해진 크기의 창 하나**뿐이다. 시스템 프롬프트·CLAUDE.md·메모리 색인·모든 메시지·읽은 파일·도구 결과가 전부 이 창에 들어간다. 창은 사람의 작업기억처럼 **어텐션 예산**이고 토큰마다 소모된다. 토큰이 늘수록 한계효용이 떨어지고, **context rot** — 토큰이 많아질수록 그 안의 정보를 정확히 회수하는 능력이 떨어지는 현상 — 이 정도 차이만 있을 뿐 모든 모델에 나타난다.[1]
> 목표는 "원하는 결과를 낼 **최소한의 고신호 토큰 집합**"이다.[1]

**원리 2 — 창 안에서도 자리가 실력을 가른다 (위치 효과)**
같은 내용도 창의 **앞·뒤에 있으면 잘 살고, 가운데에 묻히면 놓치기 쉽다**. 긴 컨텍스트에서 성능은 U자형 — 관련 정보가 맨 앞이나 맨 끝일 때 가장 높고 중간에서 크게 떨어진다(lost-in-the-middle).[7]
→ 반드시 따를 지시는 **앞(시스템 접두부)** 이나 **끝(작성 직전 재확인)** 에 둔다.

**원리 3 — 컨텍스트 먼저, 없으면 가져온다 (just-in-time)**
프롬프트를 받으면 **이미 창에 있는 것으로 먼저** 답하려 한다. 부족할 때만 파일을 끌어온다.[1]

```
1. 컨텍스트를 본다 ── 창에 있는 것으로 답 가능? ──예──> 바로 답
        │ 아니오
        ▼
2. 가져온다 ── 단, 도구마다 창 점유가 다르다:
     Read(범위 없음) ▓▓▓▓  원문 전체
     부분 Read/Grep   ▓▓    일부 라인만
     Glob             ▏     경로만
        ▼
3. 가져온 건 "이미 아는 것"에 합류 → 같은 세션에선 다시 안 찾음
```

> ⚠️ **탐색은 자동이 아니다.** 창 안에 *단서*(라우터 줄)가 있어야 "찾아봐야겠다"고 판단한다. 단서가 없으면 있는 줄도 모르고 지나친다. → 라우터를 짧고 고신호로 잘 써두는 게 중요하다.[2]

### 1B. 동적 — 여러 턴을 돌면 (루프)

**원리 4 — 창에 있는 건 계속 비용이다 (단, 종류가 다르다)**
한 번 올라온 내용은 이후 모든 턴에 다시 입력으로 들어간다. 비용은 **세 갈래**다.

| 비용 | 캐싱으로 완화되나? | 내용 |
|---|---|---|
| **처리(돈·속도)** | ✅ **크게** | 앞쪽 안정 블록(시스템·CLAUDE.md)은 프롬프트 캐싱으로 재처리 안 됨. 캐시 읽기는 기본가 ~10%, 비용 최대 90%↓·지연 85%↓[6] |
| **용량** | ❌ | 캐시돼도 토큰은 창 자리를 차지한다. 창은 유한하다 |
| **주의(품질)** | ❌ | 토큰이 많고 무관할수록 초점이 흐려진다(context rot) |

> 창을 아끼는 **진짜 이유는 "매 턴 재과금"이 아니라 유한한 용량 + 주의 분산**이다. 고정분을 앞에 두면 처리 비용은 캐시로 싸지지만, 용량·주의 비용은 그대로다.
> 실전에서 창을 가장 빨리 채우는 범인은 파일 본문이 아니라 **도구 호출 결과**(Bash 출력·grep 덤프·빌드 로그·실패 스택)다.

**원리 5 — 쌓이면 압축되고, 압축은 손실이다**
창이 차면 하네스 정책에 따라 **오래된 대화 이력**이 요약(compaction)되어 자리를 낸다. "앞부분"이 아니라 오래된 이력이며, **손실적이다** — 요약 과정에서 세부가 사라지고 되돌릴 수 없다(Read로 통째 올린 원문·도구 출력이 소실될 수 있다).[1][4]
- 예외: **프로젝트 루트 CLAUDE.md는 압축을 견딘다** — 디스크에서 재주입된다. (하위 디렉터리 CLAUDE.md는 자동 재주입 안 됨.)[5]
- 함의: **정말 지켜야 할 것을 초반 대화에만 의존하지 말라** — 이게 규칙 B(전문 고정)의 진짜 근거다.

**원리 6 — 창은 시간을 구분하지 않는다 (stale + 세션 경계)**
- **stale**: 3턴 전 Read한 파일을 그 사이 Edit하면, 창에는 **낡은 원문과 새 버전이 동시에 평평하게** 놓인다. 어느 게 최신인지 표시가 없으면 낡은 걸 근거로 답한다. (초안 v1·v2가 뒤섞이는 반복 수정 루프에서 치명적.)
- **세션 경계**: 세션이 바뀌면 창은 **초기화**된다. **세션을 넘기는 유일한 통로는 파일화된 상태** — 메모리(MEMORY 색인)·스크래치 이슈·진행 로그다. 압축의 비결정적 요약에 맡기지 말고, 넘길 것은 직접 파일에 남긴다.[1][4]

### 1C. 탈출구 — 창을 새로 열 수 있다 (서브에이전트)

**원리 7 — 서브에이전트는 별도 창이다 (컨텍스트 격리)**
서브에이전트(Task/Agent)는 **자기만의 깨끗한 창**에서 일하고, 부모에게는 **응축된 요약(often 1,000~2,000토큰)만** 돌려준다. 원문은 자식 창에서 소진되고 부모 창엔 안 쌓인다.[1]

```
부모 창 ──위임──> [자식 창: 파일1,2,3 원문 잔뜩 읽고 처리] ──요약 한 줄만──> 부모 창
```

→ 원리 4(창 비용)의 가장 강한 해법. "부분 읽기"보다 위다. 대량 탐색·파일 대량 리뷰·verbose 출력은 서브에이전트에 던져 부모 창을 지킨다. (자세히 → [Part 4](#part-4--서브에이전트-별도-창))

## Part 2. 활용 (운전법)

> 자동차 운전자가 엔진 설계까지 알 필요는 없다. **계기판 읽는 법 + 운전 습관**이면 된다. 이게 그것.

**① 점검**
- `/context` — 지금 창을 무엇이 채웠는지 항목별 토큰·%
- 상태줄 — 압축(auto-compact) 임박 경고

**② 신선도 판단 — "몇 %"가 아니라 "무관 비율"**
- 공식 경계선은 없다. 무관한 대량 원문이 큰 비중이면 창이 20%만 차도 이미 흐려진 것.
- 자문 한 줄: **"이걸 지금 지우면 이번 답이 틀려지나?"** 아니오면 노이즈.

**③ 새 세션으로 끊기 ← 가장 강력한 레버**
- 작업 성격이 바뀌면(예: 공고 분석 → 자소서 작성) 세션을 끊는다.
- 넘길 상태는 **파일로** 명시 전달(메모리·스크래치 파일).

**④ 대량 작업은 위임**
- 대량 탐색·파일 대량 읽기·verbose 출력은 서브에이전트에 던진다.
- → 지저분한 과정은 격리되고, 내 창엔 **결과 요약만** 돌아온다.

**⑤ 노이즈는 지울 수 없다 — 예방이 유일한 통제**
- 세션 중 특정 덤프만 콕 집어 삭제하는 기능은 없다(창은 덧붙이기 전용).
- 그래서 ⓐ 애초에 안 올리기(필요한 줄만 읽기) + ⓑ 세션 리셋, 이 둘뿐.

### 핵심 인사이트 — 두 레버 (라우터 = 문턱 / 세션 = 배수구)

흔한 오해: *"라우터로 둬도 한 번 올라오면 그 뒤엔 매 턴 읽으니, 전문 고정과 같은 것 아닌가?"*
절반만 맞다. **본문이 올라온 뒤 그 세션 안에서는** 라우터가 더 아껴주지 않는다(원리 4). 하지만 라우터가 이기는 지점은 "올라온 뒤"가 아니라 **"올라오기 전 / 안 올라오는 세션"**이다.

| 상황 | 전문 고정(CLAUDE.md) | 라우터 |
|---|---|---|
| 그 규칙을 **안 쓰는 세션** | 전부 상주 😰 | 한 줄만 |
| 쓰는 세션 — **쓰기 전** | 전부 | 한 줄만 |
| 쓰는 세션 — 쓴 후 | 전부 | 전부 *(여기선 동일)* |

맨 아래 칸만 같다. 라우터가 이기는 건 위 두 칸이고, "가끔 쓰는" 규칙은 정의상 위 두 칸이 대부분이다.

- **레버 1 — 분류(올리는 문턱).** 거의 매 세션 쓰는 규칙 → **전문 고정**. 가끔 쓰는 규칙 → **라우터**.
- **레버 2 — 세션 경계(내리는 배수구).** 세션은 쌓인 걸 털어내는 유일한 지점(원리 6). 작업 성격이 바뀔 때 끊으면 무관해진 본문을 벗고 깨끗한 창에서 시작한다.

> **라우터는 올리는 문턱을, 세션 리셋은 이미 올라온 걸 내리는 배수구를 담당한다. 둘은 짝이다.**
> 단, 과분할은 손해 — 새 세션마다 고정분(CLAUDE.md·색인)이 재로드되고 애써 쌓은 맥락도 날아간다. 목표는 **"한 세션 ≈ 하나의 일관된 작업 덩어리"**.

## Part 3. 구조화 규칙 (원리 → 규칙)

하네스를 꾸릴 때 새 정보를 어디에 둘지 정하는 5개 규칙.

### 규칙 A — 로딩을 3겹으로: 점진적 개방(progressive disclosure)
> 근거: 원리 1·3·4

상세를 창고에 쌓지 말고 **메타데이터(라우터)는 상시, 본문은 요청 시** 편다. Agent Skills의 공식 설계 원칙과 1:1 대응: (L1) 스킬의 `name`+`description`만 상시 로드 → (L2) 관련되면 `SKILL.md` 전문 로드 → (L3) 번들 파일은 이름만 두고 필요 시 탐색.[2]

| 겹 | 무엇을 | 어디에 | 상시 대가 |
|---|---|---|---|
| **항상 고정** | 반드시 매번 지킬 것 | CLAUDE.md 전문 (권고: <200줄[5]) | 용량 한 블록(캐시로 처리비↓) |
| **라우터 한 줄** | "이런 게 있다"는 단서 | 메모리 색인 · 스킬 `description` | 용량 = 한 줄 |
| **필요 시** | 상세 | 메모리 본문 · 스킬 본문 · 파일 | 쓸 때만 |

> **참고 지식이면 메모리/파일. 반복되는 절차(다단계·도구 순서)면 스킬.**

### 규칙 B — 절대 지킬 것은 라우터로 두지 않는다 + 재고정
> 근거: 원리 3(탐색은 자동 아님), 원리 2(위치), 원리 5(압축 손실)

- 파일 참조로 두면 "그 파일을 열자"는 판단을 깜빡할 수 있다. 매번 예외 없이 적용될 것(자소서 절대 규칙 등)은 **CLAUDE.md에 전문으로 박아** 늘 눈앞에 둔다.
- 효력을 가지려면 **매 턴 재주입되는 위치**(시스템 접두부/루트 CLAUDE.md)여야 한다.
- **재고정(re-anchoring)**: 긴 작업으로 창이 길어졌거나 지침 이탈이 보이면, 작성 직전 핵심 제약을 **끝에 한 줄로 다시 진술**한다(원리 2).

### 규칙 C — 분할은 "로딩 관문"에서만 의미가 있다
> 근거: 원리 4(올라온 뒤엔 파일 경계 소멸)

```
[파일들] ──관문(무엇을 올릴지 결정)──> [컨텍스트] ──처리──> 응답
   ↑                                      ↑
 분할이 작동하는 유일한 지점          여기선 경계 무의미(평평한 텍스트)
```

한 번 창에 올라오면 파일이 3개였든 1개였든 똑같이 취급된다. **분할의 값어치는 "일부만 올리고 나머지는 안 올리는 것" 뿐.**

> ⚠️ **`@import`로 쪼개는 것만으로는 토큰이 절약되지 않는다** — 임포트된 파일은 시작 시 전부 로드된다. 진짜 절약은 **라우터로 가리켜 필요할 때만 Read하는 구조**에서 나온다.[5]

판단은 하나로: **"이 작업이 이 내용의 *일부만* 필요한가, *항상 전체*가 필요한가?"**
- 일부만 필요한 순간이 잦다 → **나눈다**
- 항상 전체가 필요하다 → **안 나눈다** (어차피 다 올라옴, 나누면 라우터만 늘어 손해)

### 규칙 D — 모든 창 진입점을 최소화한다
> 근거: 원리 4(도구 출력이 최대 범인), 원리 7(격리)

- 큰 파일·큰 출력은 통째로 받지 말고 **범위 지정·Grep·head/필터**로 필요한 줄만.
- verbose 도구 출력(로그·탐색 덤프)은 **서브에이전트/파일 경유**로 격리.
- 이미 읽은 파일은 다시 읽지 않는다(중복 누적 방지).
- **원문 상주 vs 요약 상주**: 정확한 인용·수치가 근거인 작업(자유도 **낮음** — [자소서 절대 규칙](../CLAUDE.md))은 **원문**을 (부분) 상주/검색한다. 요약은 디테일을 잃어 사실을 흐릴 수 있다.

### 규칙 E — 루프 위생: 낡음·넘침·손실을 관리한다
> 근거: 원리 5·6

- 파일 수정 후 다시 논하기 전 **재Read** 하거나 낡은 판본을 명시적으로 폐기(stale 방지).
- 초안 v1/v2/v3는 창에 쌓지 말고 **파일에 쓰고 최신본만** 참조.
- **작업 성격이 바뀌면 새 세션**으로 끊고, 넘길 상태는 파일로 명시 전달.
- 응답이 초반 규칙을 잊기 시작하면 = 압축 임박 신호 → **자동 압축에 맡기기 전에 먼저 손실 없는 요약을 파일에 저장**하고 새 세션 시작.

## Part 4. 서브에이전트 (별도 창)

서브에이전트를 언제 쓰냐의 기준은 병렬이 아니라 **일이 얼마나 깔끔히 분해되는가(결합도)**다. 같은 도구가 전혀 다른 두 역할을 한다.

**역할 A — 단계 실행자 (일을 나눠 맡김)** ← 자소서형
단계가 낮은 결합도로 갈리고 각 산출물이 안정적 파일로 떨어질 때. 분석→`analysis.md`, 매칭→`matching.md`. 한 단계의 원자재(공고 전문)는 다음 단계에 통째로 필요 없으니 자식 창에 격리하고 파일로만 넘긴다. 문서가 드는 서브에이전트의 **첫 용도**가 정확히 이것 — "메인 창을 검색결과·로그·파일 내용으로 뒤덮을 side task는 서브에이전트에 맡기고 요약만 회수하라"[8]. 단계 사이 연속 맥락이 얕아 **오케스트레이터 한 세션으로 충분**하다.

**역할 B — 정찰병 / 맥락 방화벽 (연속 작업을 보호)** ← 기능 개발형
핵심이 하나의 연속된 맥락으로 이어져야 할 때 — 설계 결정·주변 코드·제약이 서로 얽혀(높은 결합도) 계속 들고 있어야 한다. 이걸 서브에이전트로 쪼개면 실이 끊겨 일관성이 깨진다. 그래서 **핵심 구현은 메인 스레드에 연속으로 두고**, 서브에이전트는 주변부(코드베이스 탐색·대량 파일 읽기·검증)만 흡수해 메인이 raw 덤프로 오염되지 않게 지킨다. 문서 기준 그대로다 — "여러 단계가 상당한 맥락을 공유(기획·구현·테스트)하거나 잦은 왕복·반복 다듬기가 필요하면 **메인 대화**를 써라."[8]

> **오해 교정:** "기능 개발은 서브에이전트로 컨텍스트를 관리한다"는 맞되, *핵심을 서브에이전트에 쪼갠다*가 아니라 *주변부를 떠넘겨 메인의 연속성을 지킨다*는 뜻이다. **연속성이 중요할수록 핵심은 오히려 안 쪼갠다.**

**판단 한 줄**
- 낮은 결합도 + 파일로 떨어짐 + 다음 단계가 원문 안 봐도 됨 → **단계 실행자로 위임** (자소서형)
- 높은 결합도 + 연속 맥락 필수 + 잦은 왕복 → **핵심은 메인 유지, 정찰만 위임** (기능 개발형)

둘 다 오케스트레이터는 **한 세션**이다. 차이는 세션 수가 아니라 **서브에이전트가 "일을 대신 하나(A)" vs "메인을 지켜주나(B)"**. 그리고 서브에이전트는 공짜가 아니다 — 격리로 부모 창은 아끼지만 **총 토큰·지연은 오히려 늘고**(멀티에이전트는 채팅의 ~15배 토큰[9]), 사용자에게 직접 질문(AskUserQuestion)도 못 한다[8]. 이득이 분명할 때만 더한다.[3]

> 1차 자료 확인 전문(Q1~Q5, 문서 사실 vs 추론 구분)은 → [부록](#부록--1차-자료-리서치-서브에이전트-vs-인라인).

## 실용 도구 (표·체크리스트)

### ① 배치 결정 — 새 정보를 어디에 둘까
위에서부터 답한다. 처음 걸리는 곳이 답.

```
매번 예외 없이 지켜야 하나? ─예─> CLAUDE.md 전문 (앞, 캐시됨)
반복되는 절차(다단계)인가?   ─예─> 스킬 (description=라우터)
가끔만 필요한 참고 지식인가? ─예─> 파일/메모리 본문 + 라우터 한 줄
세션 넘어 기억해야 하나?     ─예─> 메모리에 기록(색인 한 줄)
```

### ② 상주 vs 라우터 vs 검색 — 한 정보 단위 즉석 판단
| 물음 | 예이면 |
|---|---|
| 매번 예외 없이 지킨다 | **전문 상주** (CLAUDE.md, 앞) |
| 거의 모든 작업이 쓴다 | **상주** (짧으면 전문, 길면 요약본+원문 검색) |
| 작업마다 골라 쓴다 | **라우터 한 줄 + 필요 시 소환** |
| 크고, 매번 일부만 쓴다 | **검색** (Grep/범위 Read), 통째 상주 금지 |
| 정확한 인용·수치가 근거다(자유도 낮음) | **원문** 상주/검색, 요약 대체 금지 |
| 지금은 무관한데 예전에 올라와 있다 | **내린다** (새 세션/압축) — 남기면 노이즈 |

> 빠른 눈금: **자주+짧다 → 상주 / 가끔+크다 → 검색 / 항상+절대 → 전문 고정 / 대량 원문 → 서브에이전트.**

### ③ 반복 수정 루프 체크리스트 (예: 자소서 문항 다듬기)
- [ ] 파일 수정 후 다시 논하기 전 **재Read** (stale 방지)
- [ ] 초안 v1/v2/v3는 창에 쌓지 말고 **파일에 쓰고 최신본만** 참조
- [ ] 대량 출력(로그·grep·탐색)은 **서브에이전트/파일 경유**, 원문 덤프 금지
- [ ] 작업 성격이 바뀌면 **새 세션** + 넘길 것은 파일로
- [ ] 초반 규칙을 잊기 시작하면 **핵심 규칙 재고정** 또는 세션 끊기
- [ ] 자동 압축 전에 **먼저 손실 없는 요약을 파일에 저장**

### ④ 안티패턴 (실수 → 대응 규칙)
| 실수 | 대응 |
|---|---|
| 대용량 파일 통째 Read | 규칙 D |
| 절대 규칙을 파일 참조로만 둠(깜빡 위험) | 규칙 B |
| 과분할로 라우터 폭증 | 규칙 C |
| 이미 읽은 파일 재-Read | 규칙 D |
| verbose 도구 출력을 부모 창에서 받음 | 규칙 D·원리 7 |
| 참고 지식을 CLAUDE.md 전문에 박아 상시 낭비 | 규칙 A |

### ⑤ 컨텍스트 점검 — 수치로 보고, 무관 비율로 판단
| 도구 | 보여주는 것 |
|---|---|
| **`/context`** | 지금 창을 무엇이 채웠는지 **항목별** 토큰·%. 전체 = 사용 토큰 / 창 크기 |
| **상태줄** | 차오르면 "auto-compact까지 남은 %" 경고 |
| **`/cost`** | 토큰이 아닌 비용·사용량 관점 |

> 창 크기는 모델별(예: Opus 4.8 = 1M 토큰). **진짜 지표는 "몇 %"가 아니라 "올라온 것 중 이번 작업에 무관한 비율"**이다. 1M 창이 20%만 차도 대부분 무관한 덤프면 이미 흐려지고, 60% 차도 전부 고신호면 멀쩡하다.

**Messages 안을 3색으로 매긴다 — 자문: "지금 지우면 이번 답이 틀려지나?"**
| 색 | 무엇 | 처리 |
|---|---|---|
| 🟢 신호 | 지금 작업이 실제 근거로 씀(자유도 낮음이면 원문·수치) | 둔다 |
| 🟡 끝난 신호 | 과거엔 필요했으나 지금 무관(앞 단계 공고 전문·해결된 v1) | 내린다(세션 끊기·파일화) |
| 🔴 노이즈 | grep 덤프·빌드 로그·실패 스택·통째 Read 원문 | 내리고, 다음부턴 규칙 D로 애초에 안 올림 |

> ⚠️ **세션 중 "이것만 콕 지우기"는 불가 — 컨텍스트는 덧붙이기 전용.** 창을 떠나는 통로는 셋뿐: **압축**(자동·손실적) / **`/clear`·새 세션**(통째 리셋) / **자동 context-editing**(하네스 지원 시). 그래서 "내린다"는 핀셋 삭제가 아니라 ① 애초에 안 올리기(규칙 D) ② 통째 리셋 + 파일화(규칙 E)다. **지울 수 없으니 예방이 유일한 통제점.**

## 자주 헷갈리는 것 (Q&A)

**Q. "정보는 합치고, 규칙은 나눈다"가 기준인가?**
아니다. 정보냐 규칙이냐로 나누는 게 아니라, 둘 다 규칙 C 하나로 판단한다 — "일부만 필요한가, 항상 전체인가". 규칙도 항상 전체 필요면 안 나누고(자소서 절대 규칙=전문 고정), 정보도 일부만 자주 필요하면 나눈다(프로필 상세=건별 소환).

**Q. 규칙이 곧 에이전트인가? 스킬은 뭔가?**

| | 정체 | 비유 |
|---|---|---|
| **정보** | 창에 올라가는 재료(사실) | 식재료 |
| **규칙** | 지켜야 할 제약(수동적 텍스트) | 조리 원칙("태우지 마라") |
| **스킬** | "언제 → 이 정보 끌어와 → 이 순서로 처리해" 절차서 | 레시피 |
| **에이전트** | 창을 가지고 실제로 실행하는 행위자 | 요리사 |

스킬은 레시피까지고, 실제로 만드는 건 에이전트다. 스킬 자체는 아무것도 하지 않는다. 서브에이전트 = 별개의 새 창을 가진 요리사(원리 7).

**Q. 규칙이 다 올라와 있으면 안 헷갈리나?**
헷갈릴 수 있고, 그게 원리 4의 "주의 비용"이다. 원인은 "파일이 여러 개라서"가 아니라 "올라온 양이 많고 그중 지금 작업과 무관한 게 섞여서"다. 방어는 "다 올려서 다 챙긴다"가 아니라 **"지금 작업에 관련된 것만 올라와 있게"** — 라우터가 사주는 게 그것.

**Q. 라우터도 결국 매번 읽는 것 아닌가?**
그 세션 안에서만 맞다. 값어치는 "안 쓰는 세션 + 쓰기 전 구간"에 걸린다. 가끔 쓰는 규칙일수록 그 구간이 대부분이라 이득이 크다.

**Q. 리서치(탐색)를 하면 그 과정이 다 컨텍스트에 들어가나, 결과만 들어가나?**
**"누가 하느냐"로 갈린다**(원리 4 vs 원리 7).

| | 과정(도구 호출·결과·중간 추론) | 최종 응답 |
|---|---|---|
| **메인이 직접** 리서치 | 창에 **다 쌓인다** — 검색어, 읽은 파일 원문, 웹 본문, 실패 스택까지 전부 | 쌓인다 |
| **서브에이전트에 위임** | 메인엔 **안 들어간다** — 자식의 별도 창에서 소진 | 메인엔 **요약만**(≈1,000~2,000토큰) 돌아온다 |

```
메인이 직접:     [검색·Read·웹 원문 …전 과정이 메인 창에 누적] ──> 응답
서브에이전트:    메인 ──위임──> [자식 창: 전 과정 격리] ──요약만──> 메인
```

## 시간축으로 본 한 세션 (동적 요약)

```
턴1        턴5              턴12(압축)       새 세션
[규칙]     [규칙]           [규칙 유지*]     [규칙 재로드]
[파일v1]   [파일v1]         (원문 상당 소실) [파일v2만]
           [파일v2]←stale   ↑ 손실 발생
           [로그 누적]      ────────────>
 창 여유↓  ───────────────> 압축 트리거
                                            *루트 CLAUDE.md는 재주입[5]
```
(a) v1·v2 공존 = stale, (b) 로그 누적 = 도구 출력 넘침, (c) 압축 = 손실, (d) 세션 끊기 = 리셋+파일 재로드.

## 한 줄 정리

**에이전트는 유한한 창 하나로 일한다.** 고정분은 앞에 박아 캐시로 싸게 상시화하고(원리 2·4), 상세는 라우터로 가리켜 필요할 때만 펴며(원리 3), 지킬 지시는 끝에 다시 얹는다(원리 2·규칙 B). 파일은 "일부만 필요한 순간이 잦은가"로 쪼개고(규칙 C), 무관해진 큰 원문은 노이즈이니 내린다(규칙 D·E). 창이 낡고·넘치고·압축돼 손실되기 전에 **파일에 고정하고 제때 끊는다**(원리 5·6). 대량 작업은 **별도 창(서브에이전트)에 격리**한다(원리 7).

## 부록 — 1차 자료 리서치: 서브에이전트 vs 인라인

> [Part 4](#part-4--서브에이전트-별도-창)의 근거. 병렬성이 필요 없을 때 "서브에이전트 위임" vs "메인 세션 인라인"의 실제 차이를 Anthropic 1차 자료로만 확인한다. **문서가 직접 말한 사실**과 **거기서 끌어낸 추론**을 구분해 적는다. 접근일: 2026-07-14.

### 한 줄 답

**서브에이전트는 자기만의 격리된 컨텍스트 창에서 일하고 부모에게는 최종 텍스트(요약)만 돌려준다 — 부모 창을 아끼는 대신 그 요약은 선별적이다. 인라인은 모든 중간 산출물이 메인 창에 쌓이는 대신 전부 그대로 남는다.** 한 단계가 대량 원문·로그를 소비하고 그 디테일을 뒤 단계가 다시 볼 필요 없다면 → 서브에이전트, 뒤 단계가 앞 단계의 원문·정확한 문맥을 계속 참조하거나 잦은 왕복이 필요하면 → 인라인.[1][2][3]

### Q1. 서브에이전트는 진짜 별도의 격리된 창인가?

**그렇다. 명시적으로 문서화되어 있다.**
- "Each subagent runs in its own context window with a custom system prompt, specific tool access, and independent permissions."[3]
- "Each subagent starts with a fresh, isolated context window. **It doesn't see your conversation history, the skills you've already invoked, or the files Claude has already read.**"[3]
- "a fresh, separate context window. It loads CLAUDE.md and the same MCP and skill setup, but **starts without your conversation history or the main session's auto memory.**"[4]

예외 — **fork**는 "inherits the entire conversation so far instead of starting fresh… This drops the input isolation that subagents otherwise provide." 단 fork도 "its own tool calls still stay out of your conversation and only its final result comes back."[3] → 일반 서브에이전트는 입력·출력 모두 격리, fork는 출력만 격리.
또한 서브에이전트 transcript은 별도 파일에 저장돼 **메인이 압축돼도 영향 없다**: "when the main conversation compacts, subagent transcripts are unaffected."[3]

### Q2. 부모에게 정확히 무엇이 돌아오는가?

**최종 텍스트(요약)만. 중간 원문은 자식 창에 남는다.**
- "the subagent does that work in its own context and **returns only the summary.**"[3]
- "**Only the subagent's final text response comes back to your context** … The subagent read 6,100 tokens of files. **You got a 420-token result. That's the context savings.**"[4]
- "returns only a condensed, distilled summary of its work (**often 1,000-2,000 tokens**)."[1]

**손실성:** "condensed, distilled summary"[1]·"compression"[5]라는 표현 자체가 **선별**을 뜻한다(=선별적이라는 건 문서 사실). 다만 "디테일이 사라져 사실이 왜곡될 수 있다"는 **부정적 경고**는 서브에이전트 반환에 대해 문서가 명시하지 않는다 → **추론**. (메인 대화 압축에 대해선 손실 명시: "full tool outputs and intermediate reasoning are gone."[4])

### Q3. 실제로 부모 창을 아끼는가?

**그렇다. 명시적 이점으로 반복 서술된다.**
- 서브에이전트가 돕는 것 1번: "**Preserve context** by keeping exploration and implementation out of your main conversation."[3]
- "the verbose output stays in the subagent's context while only the relevant summary returns."[3]

단, **전역적으로는 토큰을 더 쓴다.** 아낀 건 "부모 창 용량"이지 "총 토큰"이 아니다: 멀티에이전트는 "use about **15× more tokens** than chats."[5] 서브에이전트 반환이 여럿이면 오히려 부모 창을 잠식: "Running many subagents that each return detailed results **can consume significant context.**"[3]

### Q4. 권하는 이유는 병렬성만인가? 명시된 단점은?

**병렬성은 여러 이유 중 하나. 컨텍스트 격리는 별개의 1급 이유로 명시된다.**
- "Use one **when a side task would flood your main conversation with search results, logs, or file contents you won't reference again.**"[3] — 병렬성 언급 없이 격리만으로 정의.
- "Common patterns" 첫 항목 제목이 "**Isolate high-volume operations**".[3] 병렬 리서치는 그 다음 별도 패턴.[3]

**명시된 단점/한계:**
- **지연.** "Latency matters. Subagents start fresh and may need time to gather context."[3]
- **왕복·공유 컨텍스트 부적합.** "Use the main conversation when: frequent back-and-forth or iterative refinement; multiple phases share significant context (planning, implementation, testing)."[3]
- **총 토큰 급증.** 채팅 대비 ~15배.[5]
- **실시간 협응 불가.** "subagents can't coordinate … LLM agents are not yet great at coordinating and delegating to other agents in real time."[5]
- **비상호작용성.** `AskUserQuestion` 등은 "depend on the main conversation's UI or session state and aren't available to subagents."[3]

**문서가 직접 준 결정표**[3]:

| 메인 대화(인라인)를 쓸 때 | 서브에이전트를 쓸 때 |
|---|---|
| 잦은 왕복·반복 다듬기가 필요 | verbose 출력이 나오는데 메인에 필요 없음 |
| 여러 단계가 상당한 맥락을 공유(기획·구현·테스트) | 특정 도구/권한 제한을 강제하고 싶음 |
| 빠르고 좁은 변경 | 작업이 자기완결적이고 요약만 돌려주면 됨 |
| 지연이 중요 | |

세 번째 선택지도 명시: "Consider **Skills** … that run in the **main conversation context** rather than isolated subagent context."[3] → **재사용 워크플로를 "메인 창에서" 돌리려면 스킬(=인라인)이지 서브에이전트가 아니다.**

### Q5. 파일 기반 핸드오프

**명시적 컨텍스트 기법으로 다뤄진다.**
- "Structured note-taking … the agent regularly **writes notes persisted to memory outside of the context window**."[1]
- "maintain **lightweight identifiers (file paths, stored queries, web links)** … dynamically load data into context at runtime."[1]
- 장기 실행 하네스: 진행 상태를 "progress file"·git 커밋으로 남겨 "**fresh context window**"에서 이어받는다.[6]

**격리와의 연결:** 서브에이전트 격리와 파일 핸드오프는 **같은 목표(메인 창을 고신호로 유지)의 두 수단**이다. 서브에이전트가 원문을 읽어 결과를 파일에 쓰고 부모/다음 단계는 그 파일을 참조하면 — 요약 반환의 손실성을 파일이 보완한다. 단 "이 둘을 결합하라"는 **명시적 한 문장**은 1차 자료에 없다 → **종합적 추론**.

### write-jaso 스킬에 대입하면

자소서는 자유도 **낮음** — 근거가 "프로필의 정확한 사실"이다. 서브에이전트는 요약만 돌려주므로, 뒤 단계가 앞 단계 원문을 계속 참조해야 하면 요약 반환은 위험하다. → **① 서브에이전트(또는 인라인)가 원문을 읽되 결과를 파일에 온전히 쓰고, 뒤 단계는 그 파일을 (부분) Read**하는 구조가 낫다. 초안 다듬기는 "frequent back-and-forth"라 문서가 대놓고 메인 대화를 권한다.[3] **결론: 읽을 원문이 크고 다음 단계가 원문 자체는 안 봐도 되는 단계만 서브에이전트로, 나머지는 인라인 + 산출물 파일화.**

### 확실치 않은 것 (문서로 확인 못 한 추론)
- "서브에이전트 요약이 손실적이라 사실을 왜곡할 수 있다" — "선별적"은 문서 사실[1][5], "그래서 정확성 훼손 위험"은 **추론**(메인 압축 손실[4]에서 유추).
- "서브에이전트 + 파일 핸드오프 결합" — 두 기법 각각은 문서가 권하지만 결합 지침 한 문장은 없음 → **종합**.
- Agent SDK 전용 반환 페이로드 세부 — 미확인(근거는 Claude Code 문서·엔지니어링 포스트).
