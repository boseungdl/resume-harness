- 참고: 발표자료 팀원소개(p23)는 이탈 전 초기 편성 기준이라 본인이 BE&FE&AWS로 남아있음 — 최종 담당(CI/CD)과 다름을 인지.

## 2. 자소서용 소재

### 지원 동기·"왜"
- **인프라 관심의 출발점**: 웹 풀스택(React/Node) 다음 단계로 "화면에 결과가 안 보이는" 클라우드/인프라를 배우던 시기의 팀 프로젝트. 개발자로 참여하면서도 **서비스가 EKS·CloudFront·Route53·GitOps·모니터링 위에서 어떻게 맞물려 돌아가는지 전체 흐름을 처음 체감**한 경험 → 이후 AWS SAA·CKA·VDI 상주로 이어지는 인프라 여정의 씨앗.
- **차별화 사고**: 시장조사(럽다·기억나무·비트윈·일상·OKIT·Highlight 등)에서 기존 서비스는 "지도에 위치를 사용자가 수동 입력"하거나 "메타데이터를 쓰면 글을 못 붙인다"는 한계를 발견 → **EXIF 메타데이터 자동 활용 + 지도 + 타임라인 + 글을 결합**하고 "중복 기능 대신 필요한 기능만 심플하게"로 좁힌 의사결정. (사람의 실제 불편에서 문제를 정의하는 성향과 연결.)

### STAR 장면

**STAR 1 — 팀 주제 수렴 (Situation~Result)**
- S: 클라우드에 치우치지 않으면서 서비스 의미도 살릴 주제를 찾아야 했고, 팀이 여러 후보(테러 예고 판별, 클러스터 동영상 인코딩, PC방 요금 정보, 파트타임 고용 플랫폼, 핫딜, 약 가격 정보 등)를 놓고 표류(회의록 3·4).
- T: 멘토(AWS 강인호 상무) 미팅 전까지 설득력 있는 주제로 좁혀야 함.
- A: 시장조사로 모수를 늘려 카테고리(여행/연애/일상)별 기존 앱을 정리하고, "위치정보 획득 방식·지도·공유·글" 축으로 비교표를 만들어 빈 칸(=EXIF 자동+지도+타임라인+글)을 차별점으로 도출.
- R: EXIF 기반 포토 다이어리로 주제 확정 → 제안서 구체화 → 최종 우수상.

## 3. 포트폴리오용 소재

### 아키텍처 개요 (텍스트)
사용자 → **Route53(DNS)** → **ACM(TLS)** → **CloudFront(CDN/캐싱)** → **AWS WAF(관리형 룰: Anonymous IP·SQL DB·Core Rule Set·Amazon IP reputation)** → **ALB(Ingress)** → **EKS 클러스터**.
- EKS: **다중 AZ · Public/Private Subnet · Node Group(On-Demand + Spot)** 구성. Private subnet에 Frontend/Backend 파드, NodePort/Ingress로 노출, **HPA + Metric Server + Karpenter**로 오토스케일(설계상). Helm으로 패키징.
- 데이터: **RDS(MySQL)** = 텍스트/메타데이터, **S3** = 이미지 저장.
- 인증: **Kakao OAuth2** (`/auth/kakao`, `/kakao/callback`) → 자체 토큰.
- API(Node App): `GET /auth/kakao`, `POST /upload`, `GET /main/data`.
- IaC: **Terraform**으로 VPC/SG/EKS/애드온/CloudFront/S3/RDS/CloudWatch 프로비저닝(콘솔+코드 병행).

### CI/CD·GitOps 흐름 (팀 아키텍처 — 상세)
- **CI**: GitHub develop repo commit/push → GitHub Actions(Setup Job → Checkout → Run tests).
- **CD**: Configure AWS credential → Login to ECR → Generate .env → **Build/Tag/Push image → Amazon ECR** → Checkout GitOps repo → **Kustomize로 image tag 업데이트** → Commit push(GitOps repo).
- **배포 동기화**: **ArgoCD**가 GitOps repo를 Webhook sync/Poll로 감지 → **EKS에 pull 방식 Deploy**(base/ + overlays/dev의 Kustomization.yaml, Deployment/Service/Ingress) → Frontend/Backend Deployment→ReplicaSet→Pod.
- **알림**: 배포 결과를 **Slack**으로 push (개발자 통지).
- **의도(설계 근거)**: kubectl 직접 배포의 자격증명 노출·수동 롤백 문제를 GitOps(선언형·Git이 단일 진실원본·pull)로 해소. GitHub Actions 채택 이유 = GitHub 통합·간단한 구축, ArgoCD 채택 이유 = K8s 리소스를 Git으로 IaC 관리(최종문서화 421~423번째 줄).
> ※ 이 CI/CD·GitOps는 **한승보 담당**(전민석 이탈 후 인수). 포폴에 "내가 구축"으로 서술 가능.

### 모니터링·보안 (팀 아키텍처)
- **CloudWatch → SNS → Lambda → Slack** 알람 파이프라인. 예: `memory_utilization_high` = pod_memory_utilization이 2분 내 2회 임계 초과 시 Slack incoming-webhook으로 [발생시간/원인/이전상태→현재상태/콘솔 바로가기] 전송(실측 캡처 2023-09-12 11:27). → **"사후 대응이 아니라 사전 감지"로 전환**한 소재.
- **WhaTap**: 쿠버네티스 에이전트(1.3.4) + 애플리케이션 에이전트(0.4.79)로 K8s/앱 메트릭 시각화.
- **WAF**: 관리형 룰 그룹으로 SQL Injection/XSS/봇/익명 IP 차단. WAF Logs → CloudWatch → SNS → Lambda → Slack.

### 핵심 기술 의사결정 (대안 비교 → 선택)
- **ACM vs Let's Encrypt**: 와일드카드 자동갱신 어려움·3개월 갱신 부담 회피, ALB·Route53 통합·자동갱신 → **ACM 채택**.
- **MySQL(RDS) vs NoSQL(DynamoDB/Mongo)**: 관계형 테이블 필요 → **RDS MySQL**.
- **이미지 저장**: DB 직접 저장 대신 **S3 + 메타데이터 분리**.
- **On-Demand + Spot 노드그룹 혼합**: 비용/가용성 균형.
- **다중 AZ 설계 / 실제 single 운영**: 구성도는 다중 AZ, 비용 문제로 실 운영은 single(멘토 조언, 회의록 7).
- **GitHub Actions CD 부분 미적용**: CD는 비용 문제로 일부 미적용(최종문서화 421번째 줄) — 한계로 정직하게 기술.

## 5. 직무별 활용 각도

- **클라우드/DevOps 지원 (강점 매칭 최상)**: EKS·GitOps(GitHub Actions·Kustomize·ArgoCD)·CloudWatch 알람·WAF·Terraform을 갖춘 프로덕션급 아키텍처에서 **CI/CD·GitOps를 직접 담당**(전민석 이탈 후 인수)한 경험을 전면 배치. CKA·AWS SAA와 함께 "kubectl→GitOps 전환, OOM→CloudWatch 사전 감지"를 브릿지. "개발을 맡다 팀원 이탈로 CI/CD를 인수받아 파이프라인을 완성"한 **주체성·빠른 습득** 서사가 강점.
- **데이터/플랫폼(DE) 지원**: `kubectl describe pod`로 OOMKilled 잡는 게 하던 일(§14-2 브릿지)의 실증 소재, EXIF 메타데이터 추출→S3 저장→시각화(수집→저장→처리→활용) 파이프라인 사고. CloudWatch 임계·알람 = Kafka lag·Airflow 스케줄러 과부하 감지와 같은 구조로 연결.
- **백엔드 지원**: React/Node/Express 3-tier, 카카오 OAuth2 + 자체 토큰, S3 이미지 업로드/메타데이터 분리, EKS 상 API 통신 트러블슈팅.
- **PM 지원**: 표류하던 주제를 시장조사·비교표로 수렴시킨 과정, Jira 간트/칸반·Notion 아카이빙·멘토 정기미팅 운영, 발표 피드백 수용·개선 사이클.
- **성장 서사(전 직무 공통)**: "웹 개발자로 참여했지만 팀원 이탈로 CI/CD를 떠맡아 인프라를 파고들기 시작한 지점" — 편력이 아니라 계층 관통의 출발점이자 필요할 때 빈자리를 메우는 주체성으로 프레이밍.
