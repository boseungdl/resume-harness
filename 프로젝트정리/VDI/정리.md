## 0. 한눈에

- **정의(한 줄)**: 이강월드 소속으로 SK브로드밴드 CloudPC/CloudX(DaaS형 VDI)를 **1차 운영팀 장애 대응 → 2차 고객사(팬택C&I) 현장 1인 상주 구축·운영·오픈**까지 관통한 경력.
- **총 기간**: **1년 2개월** (1차 2023.12~2024.07 / 2차 상주 2025.07~2025.12)
- **본인 역할**:
  - 1차: 운영팀 고객사 VOC·장애 **1차 대응** (포털 해결 or JIRA 이관) — 팀 단위 관제, 개별 고객사 담당 없음.
  - 2차: 팬택씨앤아이 현장 **1인 상주** VDI 구축·운영·오픈 — "단독 = 현장 1인 상주 + 본사 협업"(로그상 본사 책임자와 작업 분담 존재).
- **플랫폼(2차)**: **OpenStack + Ceph + Kubernetes** 위에 올라간 마이크로서비스(app-*, plat-*, mng-*, RabbitMQ, MariaDB, Redis) — Helm/Ansible(burrito) 기반 배포.
- **규모/오픈**: 사용자 **191명** 규모 서비스, 오픈 **2025.12.08**. 테넌트 3개(PCNI/PCTE/VS), Provider Network 8개.
- **고객사(플랫폼 전체)**: 경기도교육청·서울대병원·심텍·팬택C&I 등 공공·의료·제조 대형 사이트.
- **연봉**: 1차 3,600만 / 2차 3,000만. 자격증: **AWS SAA(상주 전 2025.05 취득 상태로 투입) → 상주 종료 직후 CKA(2026.02).**
- **SI 증거**: PMBOK 4단계(착수·계획·실행·통제·종료) 산출물 zip **626MB 실물 보유**.

## 2. 자소서용 소재

### 지원 동기·"왜" (about-me §5 클라이맥스와 연결)

- **전 직장이 다시 부른 신뢰 서사**: 1차 퇴사 후에도 관계를 유지 → 전 직장(이강월드)이 다시 제안해 **혼자 고객사에 상주**. "서로 이해관계가 맞았고 좋은 관계를 유지했어서." → 7축(정직·신뢰)의 실증. (about-me §6-D5)
- **"흩어진 지식이 하나로 합쳐진" 순간**: 웹→인프라→백엔드→클라우드로 흩어져 있던 학습이, **AWS SAA에서 '시스템이 유기적으로 통신하는 방식'을, CKA에서 '소프트웨어-인프라-플랫폼의 경계'를 체감**하며 실무 경험에 프레임이 끼워맞춰짐. 본인 멘탈모델: *"nginx·Kafka·Spark도 결국 하나의 소스이고 이미지로 인프라와 결합돼 나오니 내가 환경 config만 조정하면 되는 형태."* (about-me §6-D3)

- **A**: **task-scheduler·plat-ms-syncm replica를 0으로 낮춰 재생성을 막고**, openstack/cloudpc RabbitMQ의 문제 큐(tsd-nova-event-queue, task-execution-request 등)를 삭제, 정리 후 replica 원복. 안 되면 **helm rollback**으로 admin/user-portal·secret을 이전 리비전으로 복구.
- **R**: 상태 수렴·서비스 복구. → "완결성을 중시하고 되돌릴 안전장치를 갖고 배포하는 사람".

### 트러블슈팅 서사 (문제→가설→시도→해결→배움)

- **유저 포털 접속 불가 → 전체 장애 전파**: 다수 요청으로 **api-gateway의 readiness/liveness probe fail → Pod 기능 불가 → CloudPC 전체로 전파**. → "한 서비스의 헬스체크 실패가 게이트웨이를 통해 연쇄로 번지는 구조"를 현장에서 체득. (about-me §14-2 '부하 전파 = Kafka consumer lag/Airflow 스케줄러 과부하와 같은 구조' 브릿지의 실물 근거)
- **proxy-handler ↔ keepalived 의존성 문제 (CPINFRA-756)**: VIP를 쥔 keepalived와 proxy 계층의 의존 관계가 얽혀 발생한 이슈를 JIRA로 추적.
- **FQDN 적용 후**: 외부에서 유저포털 접속 시 클라이언트 다운로드 안 됨 + UI 이미지 렌더링 실패 → FQDN/DNS/프록시 경로 재점검.

### 협업·갈등·리더십 에피소드

- **다직군 조율의 실체**: 본인이 JIRA 티켓을 직접 작성해 개발팀에 넘김(원문: *"안녕하세요. 팬택 C&I 안정화 담당자 한승보입니다"* — flavor 변경 후 VM ERROR 현상 티켓). VOC(고객사)→검증→개발팀 이관을 1인이 연결하는 **가교 역할**을 실제로 수행. (about-me §6-D6 갈등 질문 4단의 '고객사↔개발팀 조율' 실물 사례)
- **본사 협업**: "단독"은 고립이 아니라 **현장 1인 + 본사(이상민M 등) 협업**. 이미지 반입·권한은 운영팀에 요청하는 구조.

## 3. 포트폴리오용 소재

### 아키텍처 개요 (텍스트)

```
[사용자 191명] ─(FQDN/SSL, L4 Proxy=SGW)→ [User Portal / Admin Portal]
      │
      ▼
  ┌──────────────────────── Kubernetes (namespace: cloudpc) ────────────────────────┐
  │ app-api-gateway · app-ms-user/resource/operation/legacy/audit/pushserver         │
  │ app-task-scheduler · app-resource-scheduler · plat-ms-syncm/sm/vid4o · plat-ci    │
  │ mng-ms-service/batch · mng-monitoring-infra(openstack/netapp exporter) · mng-log  │
  │ app-mariadb(Galera) · rabbitmq-ha-common · redis/redis-sentinel · minio          │
  └──────────────────────────────────────────────────────────────────────────────────┘
      │ (VM 생성/제어)
      ▼
  ┌──────────── OpenStack (namespace: openstack, Helm/Ansible=burrito) ──────────────┐
  │ keystone·glance·nova(api/scheduler/conductor/compute)·neutron·cinder·horizon      │
  │ mariadb-server(Galera 3replica)·rabbitmq·libvirt·openvswitch  +  Ceph(스토리지)   │
  └──────────────────────────────────────────────────────────────────────────────────┘
      │
      ▼
  [control-01~03] [compute-01~12] [sgw-01/02] [NetApp 스토리지] [방화벽/스위치 이중화]
  네트워크: bond0 VLAN 15/16/116/17/117/18/118 (테넌트/부서별 Provider Network 8개)
  NTP: Compute → Control → SGW → 외부(time.bora.net) / DNS: 168.126.63.1 (KT)
```

### 핵심 기술 의사결정 / 이해 (대안·근거)

- **폐쇄망 오프라인 배포**: 인터넷 pull 불가 → 이미지 tar 반입 → 로컬 registry(10.10.10.10:5000) push → helm이 로컬 registry 참조. "Kubernetes는 manifest만 안다 / 차트·values 변경은 helm upgrade 전엔 클러스터에 존재하지 않는 일"을 명확히 인지.
- **패치 전 영향도 판단**: "작업 간 포털 로그인 불가 / 신규 접속 불가, 기존 접속자 영향 없음" — 무중단 여부를 사전 고지.
- **DB 작업 선행 원칙**: "DB 작업 있으면 DB 먼저" — 스키마가 신 이미지와 안 맞으면 Pod가 죽으므로.
- **골든이미지 3단계**: Sysprep(초기화) → cloud-init(개별 설정) → AD 조인(회사 통합) + KMS. flavor/image에 TPM(`hw:tpm_version=2.0`)·`os_secure_boot=required`·`hw_firmware_type=uefi` 속성 필수.

### 트러블슈팅 상세 (재현 가능 수준)

- **배포 꼬임 복구 절차(실물 명령)**:
  ```
  kubectl -n cloudpc scale deploy app-task-scheduler --replicas=0
  kubectl -n cloudpc scale deploy plat-ms-syncm --replicas=0
  # openstack/cloudpc RabbitMQ 문제 큐 삭제 (rabbitmqctl delete_queue ...)
  # helm rollback -n cloudpc <release> <revision>
  # replica 원복
  ```
