## 1. 이력서용 소재

### 성과 bullet (동사+수치, 직무별 각도 표기)
> ⚠ FPS·mAP·F1은 **실측 재현 로그가 아니라 발표자료(PPT) 기재치**. 이력서에 쓸 땐 "포폴 서술 기준" 인지하고 사용. 상충 수치는 §4 참조.

- **[ML/CV]** YOLO11n을 지게차·보행자 2클래스로 파인튜닝해 **F1 0.94 이상 / Recall 1.00**(테스트 프레임 약 1,000장 샘플링, 발표자료 기준) 달성 — COCO 사전학습 대비 도메인 검출 안정화.
- **[ML/CV]** BoT-SORT 추적 파라미터 튜닝(match_thresh 0.8→0.9, track_high_thresh 0.3→0.25, NMS iou_threshold 0.4→0.03, conf 0.1→0.3, ReID 도입)으로 **ID switch·오탐·박스 뭉침(under-segmentation) 감소** — 커밋 이력으로 검증 가능.
- **[ML/CV]** 검출 실패(Lost) 구간에 **등속(CV)·CTRV 칼만 외삽**을 적용해 끊김 없는 궤적 예측 + 1초/2초 reachable zone 시각화 구현.
- **[성능]** **프레임 스킵(process_every_n=2, STRIDE)** + effective_fps 일관화로 추론 부하 절반 감소, **30fps+ 실시간 추론**(PPT 목표·서술 기준)을 겨냥해 튜닝.
- **[백엔드/데이터]** CPA(최근접점) 이차방정식 기반 2D TTC + 거리·상대접근속도 결합 **RiskScore 알고리즘 설계**, 임계값 실험(alert 0.3→0.6, 6개 후보 비교)으로 오경보/조기경보 균형점 도출.
- **[PM/협업]** **팀장**으로 5인 팀·데일리 스크럼(오전/오후 2회)·노션 TASK/스크럼 DB·trunk-based+Squash Merge Git 컨벤션 운영, **Phase 1~6** 진행 관리. 본인 **55커밋**(2위 기여자, 전체 4인+bot 중).

## 2. 자소서용 소재

### 지원 동기·"왜" (이 프로젝트를 한 이유·의미)
- 사회적 문제(=사람들 간의 문제) 해결에서 의미를 찾는 성향(about-me §5-6)과 정합: 지게차 사고는 "매년 통계가 줄지 않는 구조적 문제"이며, 개인 주의력·센서·안전관리원 모두 한계가 있다는 진단에서 출발 → **"기존 CCTV를 그대로 재활용해, 사람이 못 보는 순간을 AI가 먼저 본다"**는 접근.
- 핵심 가치 문장(대본): "핵심은 돈이 아니라 충돌을 2~3초 전 예측해 소중한 생명을 지키는 것." → 의미 지향 동기와 직접 연결.

### STAR 장면
**STAR 1 — 오탐·ID switch를 파라미터로 잡아낸 검출·추적 튜닝**
- **S**: 튜닝 전 모델이 박스·트럭을 지게차로 오탐, 가까운 두 객체를 한 박스로 묶고(under-segmentation), 회전·충돌 시 같은 객체에 새 ID를 부여(ID switch).
- **T**: 안전 시스템이므로 미탐(놓침)은 치명적 — 오탐은 줄이되 Recall은 지켜야 함.
- **A**: (1) COCO 사전학습 → 현장/Unity 영상 파인튜닝, (2) NMS iou_threshold 0.4→0.03·conf 0.1→0.3으로 가짜/뭉침 제거, (3) BoT-SORT match_thresh 0.8→0.9·track_high_thresh 0.3→0.25·ReID 도입·fuse_score 조정으로 ID 유지율 향상, (4) results.plot() 제거하고 트래커가 관리하는 strack(+Lost 외삽)만 렌더.
- **R**: 오탐·박스 뭉침·ID switch가 시각적으로 개선(Before/After), 발표자료상 F1 0.94+/Recall 1.00. *정량 감소율은 실측 로그 없음 — "커밋·Before/After로 확인 가능" 수준.*

**STAR 2 — TTC만으로 안 되는 위험을 RiskScore로 보정**
- **S**: TTC(충돌까지 시간) 단독은 정지 객체·측면통과·가까운데 느린 접근 등 사각지대에서 오판.
- **T**: "단순히 가까운 것"과 "실제 충돌 경로에 있는 것"을 구분하는 위험 점수 필요.
- **A**: CPA 이차방정식으로 TTC 계산 + 거리(정규화)·충돌반경·상대접근속도(closing speed)를 feature로 설계, 논문 4편으로 feature 근거 확보. 검증 영상+입력 CSV로 alert threshold를 0.3~0.8 6단계 실험 → **0.6 채택**(첫 alert 3.125초, 조기경고와 오경보의 균형).
- **R**: TTC가 계산 안 되는 정지 근접까지 거리 risk로 보완하는 baseline 완성. 운영 확장으로 homography 월드좌표(m)·실측 delta time 전환 설계까지 문서화.

**STAR 3 — 데이터가 없어서 만든 데이터 (Unity 합성)**
- **S**: 실제 현장 CCTV 확보 불가.
- **A**: Unity로 가상 물류창고·CCTV 시점·사고 시나리오를 합성해 학습셋 구성, YouTube 실제 사고영상으로 노이즈·우발상황 보완. real/unity 추론 진입점 분리.
- **R**: 도메인 갭을 인지한 상태에서 baseline 검증 가능한 데이터 파이프라인 확보(한계는 리스크로 명시).

### 트러블슈팅 서사 (문제→가설→시도→해결→배움)

## 3. 포트폴리오용 소재

### 아키텍처 개요 (구성요소·흐름)
- **런타임 흐름**: CCTV(RTSP) → inference(FastAPI+YOLO11+BoT-SORT+외삽+RiskScore) → WebSocket push `/internal/ingest` → backend(FastAPI, REST 13종+WS 2종) → (a) frontend로 broadcast `/ws/detect` 실시간, (b) PostgreSQL에 위험 이벤트만 INSERT. 프론트는 통계는 REST 폴링.
- **컨테이너 5종(docker compose 단일 명령)**: mock-camera(mediamtx+ffmpeg, mp4 RTSP 루프) · inference · backend · postgres(16-alpine) · frontend(Nginx+React build). 네트워크 cd-net.
- **MLOps CI/CD**: Jenkins로 주기적 검출율 점검·개선 자동반영 설계, dev/prod/ci compose override 분리.
- **핵심 설계결정(기획서 §5.2)**: FrameSlot 단일 슬롯(큐 대신 최신 1프레임 유지 → 메모리 폭발 방지), asyncio 단일 루프, asyncpg async, WS 자동 재연결(3s), PUSH_EVERY_N_FRAMES=6, RTSP_URLS 환경변수 다중 카메라, Ring Buffer(충돌 직전 10초)+Hysteresis(ON 0.6/OFF 0.3)로 chattering 방지, 사건 단위 집계 `/events/episodes`, IMM(CV+CA+CTRV) 비선형 예측(filterpy).

### 핵심 기술 의사결정 (대안 비교 → 선택 근거)
- **검출: YOLO11n** (vs Faster R-CNN 2-stage). 근거: 30fps+ 실시간·CPU/Edge 가능·작은 객체 우수. R-CNN은 ~12fps로 실시간 부적합.
- **추적: BoT-SORT** (vs SORT / DeepSORT). 근거: Motion+ReID+Camera GMC 3신호. SORT=Kalman만·ID switch 빈번, DeepSORT=카메라 보정 없음. (실제 코드 진화: DeepSORT→ByteTrack→BoT-SORT)
- **위험도: CPA(수식) + Zone(정지 객체) + IMM**. TTC 단독 한계 보정.
- **RiskScore 통합: max(TTC Risk, Proximity Risk)** — 안전 경고는 하나라도 강하면 놓치지 않는 max. (⚠ PPT는 가중합 60/25/15로 표현 → §4)
- **CD 타깃: AWS ECR** 권장(vs Docker Hub) — 내부망 속도·IAM 보안·현업 표준 어필. (서버: EC2 t3.small/medium 권장)
- **좌표계: 초기 bbox 픽셀 → 운영 homography 월드좌표(m)**, 시간축 고정 FPS → 실측 delta time.

### 트러블슈팅 상세 (재현 가능 수준)
- BoT-SORT yaml 튜닝 값 이력: track_buffer 30→20→10, match_thresh 0.8→0.9, track_high_thresh 0.3→0.25, NMS iou 0.4→0.2→0.03, conf 0.1→0.3, with_reid True, fuse_score True→False.
- 프레임 스킵: process_every_n=2, effective_fps=fps/N, 출력 fps×3/4 슬로우(외삽 화살표 거리 정확도 유지 목적).
