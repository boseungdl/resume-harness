## 1. 이력서용 소재

### 성과 bullet (동사+수치)

> ⚠ 정직성 라벨: F1 0.95+는 **합성 테스트셋 기준 측정치**. 비즈니스 KPI(Cpk 제외)는 **추정치**. 아래 각 bullet에 라벨 표기.

- **[데이터/모델]** 공개 고장 데이터가 없는 조건에서 **물리식 기반 45개 변수 × 1분 단위 × 3개월치 합성 학습셋**을 직접 설계·생성. (측정: 데이터 규모)
- **[데이터/모델]** 농장을 **4개 도메인(motor·hydraulic·nutrient·zone_drip)으로 분리한 독립 AutoEncoder(TensorFlow Keras ×4)** 를 설계·학습, **전 도메인 F1-Score 0.95+** 달성(합성 테스트셋 측정: HYDRAULIC 0.9765 / MOTOR 0.9625 / NUTRIENT 0.9549 / ZONE_DRIP 0.9573).
- **[데이터/모델]** **SHAP 기반 피처 셀렉션 자동화** — 3개 타깃 RandomForest × K-means 배경 압축으로 교집합/voting/union 3단계 피처 셋을 자동 도출, 특정 고장 모드 편향을 줄인 강건 피처셋 구성.
- **[데이터/모델]** 복원오차(MSE) 분포 기반 **6시그마 3단계 알람(주의 2σ / 경고 3σ / 심각 6σ)** + **디바운싱(n분 내 n회 초과 시 단계 격상)** 으로 헛알람·헛출동 통제.
- **[데이터/모델]** **RCA 2층 구조**(도메인 층 voting → 피처 층 per-feature 복원오차 Top-3) 구현. 특정 막힘 이벤트 1건 Local RCA에서 누수 관련 feature 감소가 AE 이상점수의 **82%** 를 차지함을 정량 입증.
- **[백엔드/인프라]** **Docker Compose 7컨테이너 오케스트레이션** 구성(FastAPI·MQTT/Mosquitto·MinIO·PostgreSQL·센서 시뮬레이터·S3 Sink·추론 API).
- **[백엔드/인프라]** **MQTT Pub/Sub + WebSocket 실시간 경로**와 **S3 Sink 10분 배치 적재 경로**를 분리한 이중 데이터 흐름 설계·구현.
- **[백엔드/인프라]** FastAPI `/predict` 단일 엔드포인트로 **4개 도메인 동시 추론 + 통합 알람 레벨 + RCA Top-3 + σ-band** 응답을 제공, 프론트가 한 채널로 소비하도록 단순화.
- **[협업/문서]** 인프라 온보딩·배포 문서화(ONBOARDING·DEPLOYMENT), `docker compose up` 한 방으로 E2E 재현 가능한 상태 구축.

## 2. 자소서용 소재

### 지원 동기·"왜" (이 프로젝트를 한 이유, 배운 의미)

- **문제의 절박함이 동기**: 관수설비는 공장 라인과 달리 "멈추면 살아있는 작물이 비가역적으로 죽는다." 다운타임 비용이 비선형이라 "사후 수리"가 아니라 "사전 감지"의 가치가 이 도메인에서 특히 크다 — 이 인과가 프로젝트 존재 이유. (about-me 6축 "의미가 엔진", 3축 "사람·현상 관심"과 연결)
- **데이터가 없는 데서 시작한 문제 정의**: "공개 고장 데이터가 없다"는 제약을 회피하지 않고, 도메인을 조사해 물리식으로 데이터를 만들어 문제를 성립시킨 것 — "왜 이렇게 되지"를 파는 기질(about-me 5장 "왜의 양날")의 건설적 발현.
- **전 계층 관통의 실증**: 데이터 생성 → 모델 → 추론 서버 → 메시지 파이프라인 → 컨테이너 배포까지 한 프로젝트에서 관통. about-me 핵심 서사("흩어진 지식이 하나로 합쳐진다")의 데이터/플랫폼 버전.

### STAR 장면

**STAR-1 — "학습 데이터가 존재하지 않는다"**
- **S(상황)**: 양액펌프 막힘 예지보전을 하려는데 공개 고장 데이터가 없고, 확보한 AI-hub 딸기 재배환경 데이터엔 펌프·유압 데이터 자체가 없음.
- **T(과제)**: 모델이 학습할 "현실에 근접한" 데이터를 확보해야 함.
- **A(행동)**: 공공데이터(딸기 개화·착과기 환경: 온도·습도·CO₂·EC·pH·관수 횟수)를 기준선으로 앵커링하고, 물리식으로 45개 변수 × 1분 × 3개월 합성 데이터 생성기를 설계. 센서를 clog 한 변수의 결정함수로만 만들면 상관계수가 ~1.0이 되어 다중공선성 필터가 다 지워버리는 문제를 발견 → **각 센서에 고유 확률성분(AR(1))을 더해 목표 상관으로 낮추는 층 구조(환경→제어→물리센서→고장주입)로 재설계.**
- **R(결과)**: 정상/이상 라벨 포함 평가셋까지 구축, 도메인 F1 0.95+ 달성. **정직성 원칙 유지 — "합성데이터 결과"로만 보고, 모델 점수 좋게 하려는 데이터해킹 금지 규칙을 문서화.**

**STAR-2 — "실시간성과 안정성을 동시에" (아키텍처 결정)**
- **S**: 센서 데이터를 실시간으로 보여주면서도, 무거운 AI 추론은 안정적으로 돌려야 함.
- **T**: 하나의 데이터로 두 요구(실시간 대시보드 / 배치 추론)를 충족.
- **A**: MQTT Pub/Sub로 생산자·소비자 분리 → 백엔드가 직접 구독해 저장 없이 WebSocket으로 즉시 전달(실시간 경로) + 같은 데이터를 S3 Sink가 10분 배치로 MinIO 적재 → 추론 엔진이 배치로 AE 추론(추론 경로). "데이터 완전성보다 실시간성·경량성"이라 판단해 Kafka 대신 MQTT(QoS로 최소 신뢰성) 선택.
- **R**: 프론트가 한 WebSocket 채널로 실시간값+추론결과를 모두 받는 단순한 구조. Docker Compose 7컨테이너로 E2E 재현 가능.

### 트러블슈팅 서사 (문제→가설→시도→해결→배움)

**(가) 희소 binary 입력이 학습을 망침**
- 문제: `is_startup_phase`(기동 직후 5분, 전체의 0.7%만 1) 를 AE 입력에 넣자 **MSE가 50배 폭발**.
- 원인/배움: 희소 binary가 재구성을 왜곡. → **"희소 binary 금지" 규칙 수립**, 운전 맥락은 연속값(`minutes_since_startup`)으로 전달. 재활성화 조건(비율 ≥10% 또는 oversampling)까지 코드 주석에 명시.

**(나) 데이터 누수(Data Leakage) 차단**
- 문제: 타깃이 `filter_delta_p_kpa`(입구-출구 압력 차)인데 입력에 입/출구 압력이 남으면 RF가 "뺄셈 수식"만 100% 학습해 진짜 원인 신호의 SHAP이 0이 됨.
- 해결: 각 타깃마다 "정답을 스포일러하는 부모 변수"를 X에서 제외하는 로직을 명문화.

**(다) 임계치가 도메인마다 안 맞음 (정직한 한계 인식)**
- 문제: 고정 σ(6시그마) 정책이 도메인별 분포(오른쪽 꼬리) 차이를 못 따라가 NUTRIENT에 오탐 94% 집중(precision 0.15), cutoff 올리면 recall 급락.
- 개선 방향: σ 고정 대신 **분포 가정 없는 방법**으로 — 라벨 있으면 PR/FAR curve로 "FAR≤5% 만족 지점"을 데이터에서 선택, 라벨 없으면 **percentile(분위) 기반** 임계. → "감으로 σ 조정"에서 "운영 제약을 데이터로 만족"으로 사고 전환.

## 3. 포트폴리오용 소재

### 아키텍처 개요 (텍스트)

- **구성요소(7컨테이너, docker-compose.yaml 정본)**:
  1. `mqtt-broker` (eclipse-mosquitto, 1883) — 센서 메시지 Pub/Sub
  2. `smart-db` (postgres:15, 5432) — 알람·점수 저장
  3. `data-lake` (minio, 9000 API / 9001 콘솔) — S3 목업, 배치 적재
  4. `sensor-simulator` — 센서값 생성·발행(restart: on-failure)
  5. `backend-hub` (FastAPI, 8080) — REST·WebSocket 허브
  6. `s3-sink` — MQTT → MinIO 10분 배치 적재
  7. `inference-api` (FastAPI, 8000, platform linux/amd64) — 4도메인 AE 추론
- **두 흐름**:
  - 실시간: Sensor → MQTT → backend-hub(구독, 무저장) → WebSocket → React
  - 배치추론: Sensor → MQTT → s3-sink(10분) → MinIO → inference(전처리·피처선택·AE추론·임계판정) → PostgreSQL → backend-hub 조회 → 프론트
- **추론 API**: `POST /predict` flat JSON(센서명→값) → 4도메인 동시 추론 → MSE→alarm level, RCA Top-3, per-feature σ-band 반환. 서버 stateless(한 요청=한 시점), 모델은 `models/*_config.json` 자동 스캔 로드.

### 핵심 기술 의사결정 (대안 비교 → 선택 근거)

| 결정 | 대안 | 선택 | 근거 |
|---|---|---|---|
| 이상탐지 모델 | 지도학습 / One-Class SVM · Isolation Forest · GMM | **AutoEncoder** | 라벨 없음+40+차원 비선형. AE는 복원오차를 피처별로 분해→RCA 자연 제공 |
| 도메인 구성 | 단일 통합 모델 | **4도메인 독립 AE** | 외부환경·작물상태로 정상범위가 변동. 도메인 분리로 오류 전파 차단·RCA 정확도↑ |
| EC/pH 화학센서 | 물리 도메인에 포함 / 폐기 | **nutrient 전용 도메인 격리** | 화학센서는 미생물막·스케일·드리프트로 오감지 잦음. 물리 도메인 오염 방지 |
| 메시지 미들웨어 | Kafka | **MQTT(Mosquitto)** | IoT 경량·실시간 우선, QoS로 최소 신뢰성. 데이터 완전성보다 실시간성 판단 |
| 필터 막힘 | 별도 AE | **룰 기반 페이지** | 학습데이터 부족·단일변수 r=0.37로 독립신호 약함 |
| 저장 처리 | 실시간 스트리밍 저장 | **메모리 버퍼 → 10분 배치 S3** | 추론 연산비용↓·안정성↑ |
| 임계 | (현재 shipped) **6시그마 μ+2σ/3σ/6σ** / (개선안) percentile·PR-curve | 3단계 σ 시행 후 분위 방식으로 이관 검토 | 아래 ⚠ 주석 |

> ⚠ **about-me와의 불일치 표시**: about-me는 "P99 분위 임계"로 요약했으나, repo의 **현재 배포 임계는 6시그마(μ+2σ/3σ/6σ)** 이고 **percentile(P99.5 등) 방식은 한계 인식 후의 "개선 방향/운영점 정합"** 으로 문서화돼 있음(03_threshold_methodology, portfolio_interview_facts). → 포트폴리오엔 "6시그마 3단계로 구현, 도메인별 분포 편차 한계를 발견해 분위(PR/FAR curve) 기반으로 고도화"로 서술하는 게 정확·강력.

### 트러블슈팅 상세 (재현 가능 수준)

- 위 §2 (가)희소binary MSE 50배 / (나)data leakage 부모변수 제외 / (다)σ고정→분위 전환. 각각 repo 코드 위치 주석에 매핑됨(feature_engineering.py·train.py·03_threshold_methodology.md).

- **Tumbling vs Sliding**: 10분 tumbling 평균은 미세 전조(순간 압력 스파이크)를 스무딩으로 소실 → sliding으로 flow_drop_rate·motor_temperature·zone1_resistance 신호 포착.

## 5. 직무별 활용 각도

- **데이터 분석가 / 애널리틱스 엔지니어 (최우선 전면 배치)**: 비지도 이상탐지(AE) + SHAP RCA + 임계 방법론(σ→분위 전환의 통계적 사고) + 정직한 측정/추정 라벨링. "라벨 없는 문제를 정의하고, 데이터를 만들고, 결과의 한계를 스스로 긋는 분석가."
- **데이터 엔지니어**: MQTT→S3 Sink 10분 배치 파이프라인 + 이중 흐름 설계 + 전처리 5단계 모듈화. P1(레이크하우스)과 묶어 "수집→저장→처리→활용" 전 구간 서사. MQTT vs Kafka 선택 근거로 미들웨어 판단력 어필.
- **백엔드 개발자**: FastAPI + WebSocket + MQTT 실시간 허브, 단일 `/predict`로 4모델 통합·stateless 설계. React 연동 단순화.
- **DevOps / 클라우드 / 인프라**: Docker Compose 7컨테이너 오케스트레이션 + 배포·온보딩 문서화 + (한계로 인정한) "CI/CD 없음→수동 배포 문서화, 자동화는 향후 과제" 정직한 서술. 현업 K8s·Docker 실무와 연결.
- **PM**: 강사 피드백 반영 사이클, ADR·SESSION_LOG 결정 이력, 2.5주 E2E 완주 일정 관리.
