# Peoch 피치 (P5) — 정리

> ✅ **역할 확정 (본인 정정 반영 — 2026-07-14)**
> 한승보의 P5 담당은 **두 축**이다:
> 1. **소득 추정 파이프라인 — 설계 참여 + 거의 공동 구현** (본인 확인). about-me §9의 "소득 추정 파이프라인" 서술은 사실. 종현과 함께 설계·구현했다(주 로직 담당은 종현, 한승보는 설계 참여·공동 구현).
> 2. **커스텀 카드 도메인 백엔드** — 카드 가입·카멜레온 카드·혜택 적용/조회, 비로그인 메인페이지(노션 API 담당자 "승보 한"으로 명확).
> - 참고: **Auth·JWT·RBAC·Redis·계약서/전자서명**은 홍의찬(의찬) 담당(기능명세·Auth 문서 저자). → about-me §9의 "인증/인가(JWT·Redis 블랙리스트)"는 본인 직접 구현이 아닐 가능성 → 이 부분만 "팀 공통"으로 처리(본인 커밋으로 재확인 권장).
> 아래 자료는 "한승보 = 소득추정 파이프라인(공동) + 카드/혜택 도메인" 기준으로 정리했다.

## 1. 이력서용 소재

> 아래는 **한승보 = 소득추정 파이프라인(공동) + 카드/혜택 도메인** 기준. JWT·Redis 인증만 팀원(의찬) 담당이라 별도 표기.

### 성과 bullet (동사+수치 중심, 직무별 각도)
- **[백엔드/데이터]** 청년 미래소득 투자 플랫폼(Peoch)의 **소득 추정 파이프라인을 설계 참여·공동 구현** — 사용자 스펙 정규화 → 기대소득 산출 → 월 지원금·환급 비율 계산으로 이어지는 핵심 평가 로직을 종현과 함께 설계·개발.
- **[백엔드]** **커스텀 신용카드 도메인 백엔드**를 담당 — 카드 가입(`/card/signup`), 카멜레온 혜택 적용(`/card/benefit`), 혜택 비율/전후 비교 조회(`/card/benefit/ratio`, `/card/benefit/compare`) API 설계·구현.
- **[백엔드]** **상점 단위 혜택 매핑 결제 엔진** 설계 — 결제 시 포스기 store_id ↔ 사용자 구독 혜택 store_id 비교로 일반결제/혜택결제 분기, 최소결제·최대할인·월 사용횟수 조건 검증(혜택 최대 3개, 월초 초기화).
- **[백엔드]** **할부·한도 로직** — 원단위 절사 + 마지막 달 잔액 보정 방식 할부(최대 12개월, 무이자), 매달 지원금 30등분 **일할 계산 한도**(서비스 시작일~말일).
- **[백엔드/데이터]** 혜택 적용 전후 금액 차이·카테고리별 혜택 비율을 **결제 이력 집계**로 시각화 데이터 제공(전월 실적 대비 비교).
- **[협업]** 7인 팀 배포 직전 TODO 20여 건(카드 신청/혜택 페이지 UI·데이터·버그) 단독 클로징 → 우수상 수상 기여.
- **[공통·팀]** 프로젝트가 **JWT 인증 + Redis, iText/PDF 계약서, 배치**를 포함한 풀스택 금융 플랫폼 — 단, 이 인증/계약서 세부 구현 담당은 팀원(의찬, §4 참조). "함께 만든 플랫폼" 각도로만 언급 권장.

### 담당 역할 상세
- 카드 안내(가입/카멜레온), 카드 서비스(디자인·내게맞는 혜택·혜택 조회), 비로그인 게스트 메인페이지.
- 프론트-백 연동(카드 이미지 로딩, 혜택 원그래프, 숫자 포맷·통화 표기), 카드 신청 최종확인 데이터 로딩 오류 수정.

### 기술 스택 (실무/학습 구분)
- **실무(프로젝트 직접)**: Spring Boot, Java, MariaDB, REST API 설계, 결제/혜택 도메인 로직, **소득추정 파이프라인(공동 구현)**.
- **학습/노출(팀 공통, 본인 직접 구현 아님)**: JWT·Redis 블랙리스트, iText PDF, Batch, Podman/Jenkins 배포.

### 정량 지표
- 시연 계정 3종(찔레/피치/칠가이 — 심사 신청/투자 신청/사용중 시나리오). **실측 성능치 없음(미측정)**.
- 혜택 3개 제한·할부 최대 12개월·한도 30등분 = 설계 규칙(자료 근거 있음).

## 2. 자소서용 소재

### 지원 동기·"왜"
- 주제 자체가 **"사람들 간의 문제(청년 실업·경제적 자립)"를 금융으로 푸는 사회적 서사** — about-me의 "사람·현상에 관심(3축)", "의미가 엔진(6축)"과 직결. 단순 CRUD가 아니라 "미래 소득을 현재로 당겨 소비를 평탄화한다"는 철학(주제선정 문서 원문)이 몰입 근거.

### ⭐ 실패·성장 서사 (about-me §6-D2 = 이 프로젝트가 원석) — 자소서 핵심 연료
- **당시의 상태(실패의 실체)**: "결과가 나빠서"가 아니라 **"내가 무엇을 하고 있는지 몰랐다."** 설계 밀도·비즈니스 반영·엣지 케이스를 중요하지 않다고 여기고 **코드만 찍어냄 = "도구에 매몰."**
- **장면 소재(자료에서 관찰된 근거)**: 이 프로젝트의 실제 개발은 **엣지 케이스·비즈니스 규칙 덩어리**였다 —
  - 혜택은 "카테고리/브랜드로 나누면 너무 복잡" → **상점 단위로 단순화**(설계 결정).
  - 퍼센트 증감 계산에서 **저번달 사용액 0 → (b-a)/b 분모 0 → +100% 버그**, "예외처리 vs 데이터 조정" 사이 팀 논의(TODO "설" 항목 원문).
  - 할부 소수점 = **원단위 절사 + 마지막 달 보정**, 한도 = **일할 계산** 같은 금융 엣지.
  - → **"그때는 이런 걸 안 중요하다 여겼는데, 실제 금융 도메인은 이 디테일이 전부였다"**는 전/후 대비를 진짜 장면으로 쓸 수 있음.
- **현재의 교훈(관점 전환)**: "같은 기능이어도 얼마나 **설계에 힘을 썼는지, 비즈니스로 녹여냈는지, 엣지 케이스를 다뤘는지**가 더 중요하다." → 이후 광고 레이크하우스(P1)의 "사후수정=엣지케이스 중심 설계" 철학으로 이어짐.
- **수상의 감정**: "기대하지 않았어서 그 순간만큼은 기뻤다. 팀이 함께 만든 결과라 더 의미 있었다."

### STAR 장면 (커스텀 카드 혜택 엔진)
- **S**: 결제 시 사용자가 구독한 혜택이 "해당 상점"에서만, 조건(최소결제·최대할인·월횟수)을 만족할 때만 적용돼야 함. 카테고리/브랜드 기준은 팀 논의 결과 "너무 복잡·혼란"으로 폐기됨.
- **T**: 혜택 최대 3개, 월초 사용횟수 초기화, 혜택 변경은 월 1회·다음달부터 적용 같은 규칙을 결제 로직에 녹이기.
- **A**: 포스기 store_id ↔ 사용자 혜택 store_id 비교 → 일치 시 benefit_id로 조건 검증 후 혜택결제, 불일치 시 일반결제. 결제/환불 상태(PAID/REFUNDED/PENDING) 관리.
- **R**: 시연에서 한도 차감·초과 시 결제 실패·할부·혜택 적용을 정상 시연, 카드명세서에서 혜택 적용 금액 확인. (정량 실측 없음)

### 성장·깨달음 한 줄
- **"기능을 만드는 것과 도메인을 설계하는 것은 다르다"** — 금융 엣지 케이스를 직접 다루며 '도구 매몰'에서 '설계·비즈니스 우선'으로 관점이 바뀐 전환점.

### 협업·리더십
- 배포 직전 개인 TODO 다수를 끝까지 클로징(뒷심 보완의 실증 각도). 팀 버그 논의에서 "예외처리 vs 데이터 조정" 같은 트레이드오프 대화 참여.

## 3. 포트폴리오용 소재

### 아키텍처 개요 (텍스트)
- **3-tier + 컨테이너 파드**: React(nginx) 프론트 ↔ Spring Boot(Java 17) 백엔드 ↔ MariaDB. 세 컨테이너를 **Podman Pod 하나(`peoch`, publish 80:80)**로 묶어 구동. Jenkins는 파드와 분리 관리(재생성 시 초기화 이슈 회피).
- **인증(팀 공통)**: JWT 발급/검증, 로그인/로그아웃, RBAC(user/admin). 로그아웃 시 refreshToken 처리. *(about-me의 "Redis 블랙리스트+만료 동기화"는 팀 자산 — 본인 구현 여부 검증 필요.)*
- **소득 평가 파이프라인(팀 공통, 종현 담당)**: 사용자 스펙(학력·학과·학점·자격증·자산·건강·성실도 등) → norm user profile → expected income → invest(월 지원금) 산출, 환급 비율(최대 20%) 계산(할인율/물가상승률 반영 논의). ML/데이터 분석 알고리즘 언급(정민영 포폴 슬라이드).
- **카드/혜택 도메인(본인)**: 카드 가입→디자인→혜택 구독(카멜레온)→결제(상점단위 혜택 매핑)→명세서/집계. 지원금 한도 30등분 일할 계산.

### 핵심 기술 의사결정 (대안 비교 → 선택)
| 결정 | 대안 | 선택·근거 |
| --- | --- | --- |
| 혜택 적용 단위 | 카테고리/브랜드 vs **상점(가맹점) 단위** | 상점 단위 — 카테고리/브랜드 분리는 "너무 복잡·혼란", 규칙 단순화 |
| 할부 소수점 | 반올림/균등 vs **원단위 절사+마지막 달 보정** | 총액 일치 보장(3,300/3,300/3,400 예시) |
| 지원금 한도 | 월정액 vs **일 30등분 일할 계산** | 서비스 시작일~말일 비례로 공정 |
| 배포 | (수업 환경) Podman+Buildah, Jenkins 별도 | Docker 대신 podman/buildah, Pod 묶음·Jenkins 분리 관리 |
| 상환 방식(제품 컨셉) | 고정 원리금 vs **소득 비례 유동 상환** | 청년 리스크 분담(PIPO 철학) |

### 트러블슈팅 상세 (재현 가능 수준)
- **증감률 100% 버그**: 전월 사용액 0일 때 (b−a)/b = b/b = 100%. 해결안 = 예외처리 vs "전월 0이 안 되게 시연 데이터 조정" 트레이드오프(복잡도 고려해 후자 논의). → 엣지 케이스 인식 사례.
- **배포 한글 깨짐**: DB bash에서 `export LANG=C.utf8` 로케일 설정.
- **Jenkins 내부 UI 접근**: `ssh -L 9090:192.168.0.172:9090 root@…` 포트포워딩 터널링.
- **nginx 설정 수정**: `podman cp`로 default.conf 꺼내 nano 수정 후 되넣고 `nginx -s reload`.

## 5. 직무별 활용 각도

- **백엔드 개발자 (1순위)**: 카드/혜택 **결제 도메인 로직**(상점단위 매핑·할부·한도·엣지케이스)을 STAR로 전면 배치. "도구 매몰→설계·비즈니스 우선" 성장 서사가 강력. 팀 공통 JWT/소득예측은 "함께 만든 금융 플랫폼" 배경으로만.
- **PM / 기획**: 주제 선정(후보 6종→PIPO 수렴), 배포 직전 TODO 클로징, 팀 버그 트레이드오프 논의 = 실행·조율 소재.
- **데이터 분석/DE**: 중간 — **소득추정 파이프라인(스펙 정규화→기대소득→환급률) 공동 설계·구현** 경험을 "비정형 사용자 데이터를 평가 지표로 변환한 로직" 각도로 활용 가능. 다만 주력은 P1·P3.
- **금융 IT/SI**: 신한DS 우수상 + 금융 도메인(상환·한도·할부·혜택) 규칙 설계 경험 = 도메인 적합성 어필.
