[2차] 팬택씨앤아이 현장 1인 상주 - K8s 기반 VDI 플랫폼 구축·운영·오픈
      2025.07 ~ 2025.12   |   현장 1인 상주(+본사 협업)
────────────────────────────────────────────────────────────
● 개요 : OpenStack+Ceph+Kubernetes 위 마이크로서비스(app-*, plat-*, mng-*, RabbitMQ, MariaDB,
         Redis)로 구성된 DaaS형 VDI를 고객사 현장에서 단독 상주로 구축·운영하고,
         사용자 191명 규모 서비스를 오픈(2025.12.08). 테넌트 3개 / Provider Network 8개.

1. 플랫폼 운영 (OpenStack + Ceph + Kubernetes)
   - K8s 위 마이크로서비스 상태 점검·이슈 대응 (kubectl get/describe/logs로 CrashLoopBackOff·
     Error·ImagePull·Pending Pod 3레벨 조회 루틴화)
   - 컨테이너/데몬을 프로세스·포트·리소스 관점으로 이해하고 환경 config 조정·상태 점검
   - 현장 1선 장애·서버 대응 (여러 시스템이 얽힌 복잡한 VDI 환경)

2. 배포·패치 (폐쇄망 오프라인 환경)
   - 인터넷 pull 불가한 폐쇄망에서 이미지 tar 반입 → 로컬 registry(10.10.10.10:5000) push →
     override values 수정 → helm upgrade → Pod 재기동 절차로 플랫폼 패치 PCNI-2.2.1~2.2.5 적용
     (인사연동·근태·포털 커스터마이징·OTP/MFA·SMTP 암호화 등)
   - 패치 시 DB 마이그레이션 직접 수행 (MariaDB에 kubectl exec 접속, 테이블·컬럼·메뉴/권한 데이터·
     이벤트 스케줄러 생성 SQL 적용) - "DB 작업 선행" 원칙 준수
   - FQDN·SSL 인증서 배포 config 적용, JIRA 기반 이슈 트래킹·이관(CPINFRA-756 등)

3. 이중화 · 서비스 오픈 (Go-live)
   - 전원·네트워크·서비스 이중화 테스트 전 영역 수행 (스위치·control/compute 서버·스토리지(NetApp)·
     방화벽·SGW)
   - VLAN(bond0.15~118) 구성·검증, SGW(L4 Proxy) NTP 서버 구성(Compute→Control→SGW→외부),
     chrony 동기화
   - 사용자 191명 규모 VDI 서비스 구축·안정화·오픈 (2025.12.08) [측정]

4. 이미지·자산 관리
   - 골든이미지 제작·Sysprep → cloud-init → AD 조인 → KMS 3단계 배포 체계 수행
   - OpenStack 이미지/플레이버에 TPM(hw:tpm_version=2.0)·os_secure_boot·uefi 속성 설정

5. 표준 문서화 (SI 전체 사이클 · PMBOK 5단계)
   - 요구사항정의서·WBS·네트워크/방화벽/테넌트 설계서·이중화 테스트 계획/결과서·CloudX 기능검증·
     골든이미지 관리대장·변화관리 방안·주간보고(5~16주차)·완료보고서 등 산출물 참여

● 트러블슈팅 (대표)
   - 유저 포털 접속 불가 → 다수 요청으로 api-gateway probe fail → Pod 기능 불가 → 전체 전파.
     헬스체크 실패가 게이트웨이를 통해 연쇄 전파되는 구조를 현장에서 규명·대응
   - 배포 꼬임 복구: 관련 deploy replica=0으로 재생성 차단 → RabbitMQ 문제 큐 삭제 → helm rollback
     으로 이전 리비전 복구 → replica 원복

● 협업 : 고객사·개발팀·사업·영업과 다이렉트 소통, VOC·테스트 이슈를 JIRA 티켓으로 직접 작성·이관.
         "단독"의 의미 = 현장 1인 상주 + 본사(운영팀) 협업.


────────────────────────────────────────────────────────────
[1차] 운영팀 - 고객사 VOC·장애 1차 대응
      2023.12 ~ 2024.07   |   팀 단위 관제
────────────────────────────────────────────────────────────
● 개요 : SK브로드밴드 CloudPC/CloudX 고객사 VOC·장애를 1차 대응하고 포털 해결 또는
         시스템엔지니어/개발자에게 JIRA 이관하는 운영팀 업무.

1. 고객사 VOC·장애 전화·메일 1차 대응 → 포털 해결 or JIRA 이관·회신 (팀 단위 관제)
2. VM 대량 회수로 인한 생성 지연 유형 장애 대응 참여
3. 장애보고서 분석으로 플랫폼 장애 전파 구조(스케줄러 대기열·메시지 큐·스토리지 리밸런싱) 학습

※ 참고: 상주(2차) 시절 본인이 직접 대응한 장애 기록은 원본이 삭제되어, 유형이 유사한 과거
   운영팀 장애 보고서를 참고자료로 보관 중. → 면접·자소서에서는 "대응한 장애 유형·전파 메커니즘"
   중심으로 서술하고, 특정 날짜·사건을 본인 것으로 인용하지 말 것.
