## 2. 자소서용 소재

### 지원 동기·"왜" (이 프로젝트를 한 의미)
- 신한DS Java 교육의 팀 프로젝트로, **웹 풀스택→클라우드 이후 "백엔드를 팀 규모로 처음 붙어본" 경험**. about-me §5-5 / D2 기준, 이 시기 본인은 "인프라·자격증에 신경 쓰다 포지션이 어정쩡했고", "**내가 무엇을 하고 있는지 몰랐다**"는 실패 감각을 남긴 프로젝트다. → 이후 "같은 기능이어도 설계·비즈니스 반영·엣지 케이스가 더 중요하다"는 관점 전환의 **출발점**(P5 피치·P1 레이크하우스 설계 철학으로 연결).
- 도메인 자체(반려견 여행)보다, **게시판이라는 흔한 기능을 '조회 성능·이미지 저장 위치' 같은 구조 관점에서 다시 본 것**이 본인 "구조를 먼저 보는 사람"(정체성 1축) 성향과 맞물린 지점.

### STAR 장면

**STAR-1. 게시글 상세 조회의 반복 쿼리(이미지·좋아요)**
- **S**: 게시글 1건에 사진 여러 장 + 좋아요 수를 함께 보여줘야 함. 명세 단계 메모에 "**두 번 나눠서 쿼리 나누는 게 나은지 한 번에 조인하는 게 나은지 확인하기**"라는 미결 고민이 실제로 남아 있음(Community-3 문서).
- **T**: 게시글마다 이미지·좋아요를 따로 조회하면 목록·상세에서 반복 쿼리가 늘어남.
- **A**: MyBatis `resultMap`에 `<collection>`을 두어 이미지(1:N)를 매핑하고, 좋아요는 `LEFT JOIN (SELECT post_id, COUNT(*) ... GROUP BY post_id)` 서브쿼리로 집계해 **게시글 상세를 이미지·좋아요까지 한 쿼리로 조회**(postMapper `findPostById` / `PostVOResultMap`).
- **R**: 이미지·좋아요 반복 조회 제거. **단, 댓글은 별도 쿼리(`getComment`)로 남김** → "전부를 한 방에 합치는 것"이 항상 답은 아니라는 판단도 함께 경험(§4 정정).

**STAR-2. 이미지 저장 위치를 DB에서 S3로 분리**
- **S**: 게시글 작성 시 사진 필수(요구사항). 이미지를 DB에 넣으면 용량·조회 부담.
- **T**: 이미지 저장 전략 결정.
- **A**: **AWS S3에 파일 업로드하고 DB에는 URL만 저장**하는 구조 채택. `post_images` 테이블을 후속 추가(2024.12.26 테이블 추가 기록), `util/awsS3` 유틸 패키지 구현.
- **R**: DB에는 메타데이터만, 원본은 오브젝트 스토리지로 분리 — 이후 데이터/인프라 학습(S3, MinIO Sink 등 P1·P3)과 개념적으로 연결되는 첫 오브젝트 스토리지 경험.

**STAR-3. Ajax 응답 500 — LocalDateTime 직렬화 오류**
- **S**: `main→reviews` 이동 시 `@RestController` JSON 응답이 계속 500. 소통·로직엔 문제 없어 원인이 안 보임.
- **T**: 원인 규명.
- **A**: **Tomcat Localhost Log**를 직접 확인 → `InvalidDefinitionException: Java 8 date/time type LocalDateTime not supported by default`. Jackson이 Java8 시간타입을 기본 직렬화 못 함을 파악. `jackson-datatype-jsr310` 의존성 추가 + `JavaTimeModule`을 `ObjectMapper`에 등록, `@JsonFormat`으로 포맷 지정.
- **R**: 직렬화 오류 해소. **"로그를 끝까지 읽어 근본 원인(변인)을 찾는다"**는 about-me의 '왜를 파는 사람' 기질의 소규모 실증.
- (주의: 노션상 이 트러블슈팅 문서 작성자 정보는 없으나 review 도메인 DTO가 등장 → 본인 단독 여부 불확실. "팀 트러블슈팅 문서" 수준으로 인용 권장.)

### 트러블슈팅 서사 (문제→가설→시도→해결→배움)
- 위 STAR-1/2/3이 그대로 서사. 공통 배움: **"기능이 되는 것"과 "구조적으로 다루는 것"은 다르다** — 조회 쿼리 합치기·이미지 저장 위치·직렬화 설정처럼 눈에 안 보이는 결정이 품질을 가른다.

### 협업·리더십 에피소드

## 3. 포트폴리오용 소재

### 아키텍처 개요 (텍스트)
- **구성**: 브라우저(JSP + jQuery/Ajax) ↔ Spring MVC(Controller→Service→Repository/MyBatis) ↔ MySQL. 이미지 원본은 **AWS S3**, DB엔 URL. 실시간(플래너 마우스·채팅)은 **WebSocket**. 로그인은 **OAuth2(Kakao/Google/Naver)**. 세션 캐시/보조에 Redis. 배포는 **EC2 + Jenkins**.
- **도메인 분리(패키지)**: `member`, `plan`, `place`, `post`(커뮤니티=본인), `comment`(post 하위), `photocard`, `review`, `notification`, `chat`, `util/awsS3`, `config`.
- **커뮤니티 흐름(본인)**: 게시글 목록(하트순 LIMIT 20) → 상세(이미지 collection + 좋아요 집계 JOIN, 댓글 별도 조회) → 작성 시 S3 업로드 → 좋아요/댓글/검색/카테고리/정렬 → 게시글-플랜 매핑.

### 핵심 기술 의사결정 (대안 비교 → 선택 근거)
| 결정 | 대안 | 선택 | 근거 |
| --- | --- | --- | --- |
| 게시글 상세 이미지·좋아요 조회 | 게시글당 이미지/좋아요 개별 쿼리(N+1) vs 조인+resultMap collection | **collection + LEFT JOIN 1쿼리** | 이미지 1:N, 좋아요 count를 한 번에. 반복 조회 제거 |
| 댓글 조회 | 게시글 쿼리에 함께 조인 vs 분리 | **분리 쿼리(`getComment`)** | 댓글은 별도 목록/페이징·members 조인이 필요 → 게시글 조회와 분리가 단순·명확 (합치기가 늘 정답 아님) |
| 이미지 저장 | DB BLOB vs S3+URL | **S3 + URL(메타데이터)** | DB 부하↓, 정적 파일은 오브젝트 스토리지가 적합 |
| 팔로우 관계(팀 DB설계) | 컬럼 배열 vs 연결 테이블 | **`relationships` 연결 테이블 + `(from_email,to_email)` 복합 PK** | 다대다 팔로우/중복 방지 (※ API 구현 담당은 유성훈, §4) |
| 좋아요 | 카운트 컬럼 vs 연결 테이블 | **`likes(member_email,post_id)` 복합 PK** | 사용자당 1회 보장 + FK 제약 |

### 트러블슈팅 상세 (재현 가능 수준)
1. **LocalDateTime JSON 직렬화 500** — 증상: `@RestController` JSON 500 / 로그: `InvalidDefinitionException: Java 8 date/time type java.time.LocalDateTime not supported by default` / 원인: Jackson 기본 모듈이 JSR-310 미지원 / 해결: `jackson-datatype-jsr310` 의존성 + `mapper.registerModule(new JavaTimeModule())`(`JacksonConfig`) + `@JsonFormat(pattern="yyyy-MM-dd HH:mm:ss")`. (근거: InvalidDefinitionException.md)
2. **Spring DB 연결 Access denied** — `CannotGetJdbcConnectionException ... Access denied for user 'root'@'121.x'` / 원인: root 접근 권한이 localhost로 제한 / 해결: 외부 IP(`%`)로 권한 확장. (근거: DB 트러블슈팅/Spring에서 DB연결 에러.md) — **보안상 실서비스엔 부적절한 임시조치**임을 함께 언급하면 좋음.
3. **JSP에서 이미지 깨짐** — 원인: `WEB-INF` 내부는 외부 접근 불가 / 해결: `WEB-INF` 밖 `webapp` 아래로 이미지 이동 + context path 변수(`${path}/resources/...`). (근거: JSP에서 이미지 보기.md — 작성자 김신영)
4. **`can not resolve <변수>`** — JSP 템플릿 리터럴 내 JS `${var}`를 JSP EL로 오해석 → `\${...}` 이스케이프. (근거: can not resolve.md — 작성자 김신영)
5. **`spring_web fragment 중복`** — 빌드 방식 변경 누적 → Maven `clean`. (근거: 해당 md)

> 재현 가능한 "본인 트러블슈팅"으로 가장 안전한 건 **1번(LocalDateTime)**과 커뮤니티 도메인 관련 조회 설계. 3·4번은 작성자가 김신영 → 본인 것으로 쓰지 말 것.

### 불확실/검증 필요 항목 (⚠️ about-me와의 불일치 포함)
1. **⚠️ "게시글–댓글 N+1 → resultMap JOIN 1회로 축소"(about-me §8, §9-P6)는 부정확.**
   실제 코드(postMapper `findPostById`)는 **이미지(collection)와 좋아요(집계 JOIN)를 한 쿼리로 합쳤고, 댓글은 별도 쿼리(`getComment`)로 유지**. 자소서·이력서엔 "게시글 상세의 **이미지·좋아요 반복 조회를 resultMap+JOIN 한 쿼리로 축소**"로 정정해 쓸 것. "댓글"을 예로 들면 코드와 어긋남.
2. **⚠️ 팔로우 다대다(연결테이블+복합유니크)는 본인 구현이 아님.** DB엔 `relationships (from_email,to_email) 복합 PK`로 존재하나, 팔로우 API(Member-20~23)는 **담당자 "성훈 유"**. 본인 자산으로 쓰지 말 것 → "팀 DB 설계 결정" 또는 "좋아요 연결테이블(likes 복합 PK)"로 대체 권장(좋아요는 본인 도메인).
3. **수상 = 수상작 확정**(본인 정정). 기존 about-me 수상 표엔 누락. **등급·주최명은 자료에 없어 본인이 채워야 함** → about-me §10에 추가 필요.
4. **본인 개인 커밋 수 미확인**: about-me의 다른 프로젝트처럼 "본인 커밋 N개"가 없음. repo가 팀 계정(kseenyoung)이라 개인 기여도 정량화 불가 → "커뮤니티 도메인 담당"으로 질적 표현.
5. **LocalDateTime 트러블슈팅 작성자 불명**: 본인 단독 여부 불확실. "팀 트러블슈팅"으로 안전하게 인용.
6. **성능 before/after 수치 없음**: N+1 개선의 정량 효과(쿼리 수·ms) 측정 기록 없음 → "실측치 없음" 명시.
7. **기간 종료일 불명확**: 착수 2024.12.04 확인, 종료·발표일은 자료상 특정 안 됨(교육 일정상 2025.01~02 추정).
8. **OAuth Naver 포함**: about-me는 Kakao/Google만 언급하나 yml엔 Naver도 설정됨(활성 여부는 미확인).

## 5. 직무별 활용 각도

- **백엔드 개발자 (주력)**: 커뮤니티 도메인 CRUD + **MyBatis resultMap/collection·JOIN으로 이미지·좋아요 조회 최적화** + **S3 이미지 분리** + **Jackson 직렬화 트러블슈팅**을 전면 배치. "설계·엣지케이스보다 코드만 만들던 나 → 조회 구조를 다시 본 전환점"으로 성장 서사와 엮기. (about-me §3 백엔드 매핑의 "댕동 CRUD·N+1·MyBatis" 항목이 여기 근거 — 단 §4-1대로 표현 정정.)
- **PM / 프로젝트 관리**: **데일리 스크럼·KPT·WBS·Git-flow 진행 관리 참여** + 도메인별 명세서 담당자 분배 체계. **"팀장"이 아님을 정확히** — "진행관리 주도적 참여"로. (about-me §3 PM 매핑 "댕동 스크럼·KPT·WBS"와 일치.)
- **데이터 엔지니어 / 인프라 (보조)**: **S3 오브젝트 스토리지 첫 경험**을 P1(레이크하우스 S3)·P3(MinIO Sink)로 잇는 "저장 계층을 분리해 다루기 시작한 지점"으로 짧게. 깊이는 없으므로 주력 아님.
- **주의**: 이 프로젝트는 **깊이보다 "팀 백엔드 첫 경험 + 구조 관점의 각성"**이 핵심 가치. 기술 자랑보다 **성장 서사(D2 실패 → 관점 전환)**의 연료로 쓰는 게 진실에 부합.
