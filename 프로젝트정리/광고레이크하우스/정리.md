## 1. 이력서용 소재

### 성과 bullet (동사 + 내용) — 직무 각도 표기

> ⚠ 실측 성능 지표(쿼리 지연·비용 절감률 등)는 랩 규모라 **미측정**. 아래는 "구현·설계한 것" 중심. 억지 수치화 금지.

- **[DE]** Criteo 광고 이벤트(30일 16.4M행)를 **Kafka 3토픽(impressions/clicks/conversions) → Spark Structured Streaming → S3 Bronze**로 수집하는 스트리밍 파이프라인을 Docker Compose 환경에서 구축·실행.
- **[DE]** 전환 지연(late arrival, P95 3일·최대 30일) 문제를 **event_id 기준 멱등 MERGE INTO**로 해결 — 같은 event_id 행에 뒤늦게 도착한 전환을 수렴시켜 재처리 시 중복 0 보장.
- **[DE]** MERGE를 **2개 Airflow task로 분리** 설계: `insert_new_events`(신규만 LEFT ANTI JOIN INSERT, 경량) + `update_late_arrivals`(지연 도착 전환만 MERGE-WHEN-MATCHED, 저빈도·고비용) → 그래프 뷰에서 INSERT/UPDATE 분기를 분리 관측.
- **[DE]** **Medallion 3계층**을 구간별 특성에 맞게 설계: Bronze=수집시각 파티션(쓰기 최적·원본 보존, Iceberg 밖 append), Silver/Gold=이벤트시각 파티션 + Iceberg(정제·분석 최적).
- **[DE/DevOps]** **Airflow DAG 3종**으로 오케스트레이션: raw→Silver, Silver→Gold를 매시간 적재하고, 유지보수(compaction·expire snapshots·orphan 정리)를 별도 주기로 분리. BashOperator + `docker exec`로 spark-iceberg 컨테이너에 위임. *(정확한 cron 값은 §4 확인 필요)*
- **[DE]** 잦은 업데이트(전환 지연 소급 갱신) 워크로드에 **Merge-on-Read(MOR)** 채택 — 쓰기 지연(commit latency)·write amplification 최적화. `format-version=2`, `write.{update,merge,delete}.mode=merge-on-read`.
- **[DE]** 스몰파일 관리: Silver/Gold는 `rewrite_data_files`(target≈128MB, min≈50MB, min-input-files 5, partial-progress)로 병합, Bronze(Iceberg 밖)는 별도 머지잡. position-delete 재작성·snapshot 만료를 운영 배치로 분리.
- **[DE]** 읽기 성능 최적화로 processed 테이블에 **z-order 정렬**(`zorder(campaign, uid)`) 적용, 증분 MERGE는 **5일 슬라이딩 윈도우**(`WHERE event_date > current_date() - INTERVAL 5 DAYS`)로 소스 범위 제한.

## 2. 자소서용 소재

### 지원 동기·"왜" (이 프로젝트를 한 이유)
- **설계 흥미 × 인프라 자산 × 분산처리 탐구욕의 수렴점**. 취준생 입장에서 돈 내며 대용량 인프라를 경험하기 어려운 갈증을, "설계를 하면서 동시에 데이터를 다루고 분산환경(CKA)·Spark 분산처리를 깊게 공부"하는 방향으로 풀고자 이 프로젝트를 택함. (about-me §6-D5)
- **관점 전환의 결과물**: 과거 프로젝트(Peoch)에서 "코드만 만들고 설계·비즈니스·엣지 케이스를 경시했던" 실패를 반성. 그 반성이 "같은 기능이어도 얼마나 설계에 힘을 썼는지, 엣지 케이스를 다뤘는지가 더 중요하다"는 신념으로 이어졌고, **전환 지연(엣지 케이스) 중심 설계**가 그 신념의 실물 구현. (about-me §6-D2)
- **"왜"를 파는 기질의 생산적 활용**: 단순히 랩을 따라 돌리지 않고 "왜 이 파티션인가, 왜 Iceberg인가, 왜 MOR인가"를 요구사항까지 역추적해 문서화(설계고민 문서). AI를 로직 검증·데이터 정리 보조로 쓰며 "왜 매몰"을 속도로 보완.

### STAR 장면

**S1. 전환 지연 → 파티션 전략 전환**
- **Situation**: 전환(conversion)은 클릭 후 늦게 도착(P95 3일, 최대 30일). 발생 시각(event time)으로 파티션하면 과거 파티션을 계속 다시 열어 갱신해야 함.
- **Task**: 지연 도착을 감당하면서 쓰기·재처리 비용을 낮추는 저장 구조 설계.
- **Action**: Bronze는 **수집 시각(ingest/ETL time) 파티션**으로 전환(늦게 온 데이터도 재파티션 없이 현재 파티션에 append, 원본 보존·쓰기 최적). Silver/Gold는 분석 편의를 위해 **이벤트 시각 파티션**으로 두고, MERGE 시 늦게 온 전환을 올바른 event_date 행으로 수렴.
- **Result**: 수집(쓰기)과 분석(읽기)의 요구를 분리해 각 구간을 최적화. (실측 비용 수치는 미측정)

**S2. event_id 멱등 MERGE (2-pass)**
- **Situation**: Kafka 컨슈머 at-least-once + Spark 마이크로배치 재시도로 **같은 이벤트가 중복** 유입 가능. 정제 없이 집계하면 전환 카운트가 뻥튀기됨.
- **Task**: 중복 없이, 지연 도착 전환을 같은 행에 반영하는 정제 로직.
- **Action**: `MERGE INTO ... ON t.event_id = s.event_id` — 신규는 INSERT, 기존 event_id의 늦은 전환/클릭은 UPDATE. 디버깅 편의를 위해 변경 컬럼만이 아니라 전체 컬럼을 업데이트해 추적성 유지. 운영에선 이를 **2개 task로 분리**(대부분 저렴한 INSERT / 드물지만 무거운 UPDATE).
- **Result**: 재처리해도 결과 동일(멱등). snapshot/history로 상태 변화 추적. → **면접 방어 자산: 직접 돌려봐서 설명 가능.**

**S3. MOR vs CoW 선택**
- **Situation**: 전환 지연으로 **같은 파티션을 자주 업데이트**. CoW는 매 commit마다 데이터 파일 전체를 재작성(write amplification)해 매시간 MERGE에 부담.
- **Task**: 잦은 소량 업데이트에 맞는 쓰기 모드 선택.
- **Action**: **Merge-on-Read** 채택(`format-version=2`, `write.*.mode=merge-on-read`) — commit 시 position-delete 파일만 추가해 쓰기 지연 단축. 대신 읽기 시 delete 병합이 필요하고 **주기적 compaction + delete 재작성이 필수**라, `daily_maintenance`를 DAG에 포함.
- **Result**: 쓰기 지연/읽기 성능/운영 부담의 트레이드오프를 워크로드 특성으로 정당화. (Gold처럼 읽기 중심이면 CoW/RDB가 낫다는 반례까지 인지.)

### 트러블슈팅 서사 (강의에서 배운 운영 실패 패턴 — 본인이 "인지하고 설계에 반영"한 수준)
- **동시 쓰기 오염**: 스트리밍과 배치가 같은 파일을 건드릴 때, 전통 카탈로그는 락이 약해 success로 보여도 데이터가 오염됨 → Iceberg 낙관적 커밋(base snapshot 비교 후 다르면 retry) + **시간대 분리 설계**(적재 vs 매니지먼트)로 충돌 회피.
- **스몰파일 폭증**: 30초 트리거 스트리밍은 하루 수천 스냅샷·수천 마이크로파일 → S3 LIST/GET 비용↑, 쿼리 플래닝 지연 → `rewrite_data_files` compaction + snapshot expire로 대응. "added-data-file이 평소 3인데 갑자기 100"을 스몰파일/파티션 이상 신호로 읽는 **스냅샷 서머리 모니터링** 관점 학습.
- **고아파일**: 원자성 작업 실패·수동 업로드·compaction 잔재로 발생 → remove_orphan_files를 별도 배치로. "파일 수가 수십→수천이면 expire/compaction 비정상 종료 신호이니 무작정 삭제 전 점검"이라는 운영 판단 학습.

### 성장·깨달음 한 줄
- **"결과물보다 데이터가 흐르는 구간의 설계와 운영 가시성이 엔지니어의 실력이다."** — 랩을 따라 돌리는 데서 멈추지 않고, 각 결정을 요구사항까지 역추적해 "왜"로 설명 가능하게 만든 것이 이 프로젝트의 핵심 소득.

## 3. 포트폴리오용 소재

### 아키텍처 개요 (구성요소·흐름)
```
[Criteo 이벤트]                     ┌──────────── AWS ────────────┐
prepare_criteo_data.py             │  S3 (Object Storage)         │
   → ad_events.csv                 │  Glue Data Catalog           │
        │                          │  Athena(쿼리) · QuickSight(BI)│
        ▼                          └──────────────────────────────┘
  kafka_producer.py
        │  (3 topics: impressions / clicks / conversions)
        ▼
      Kafka  ──(버퍼·디커플링)──►  Spark Structured Streaming
                                        │  (append, checkpoint)
                                        ▼
   Bronze (수집시각 파티션, append, Iceberg 밖 / 원본 보존)
                                        │  MERGE INTO (event_id 멱등, 2-pass)
                                        ▼
   Silver (Iceberg v2, MOR, event_date 파티션, 중복제거·정제)
                                        │  Incremental 집계(영향 파티션만)
                                        ▼
   Gold  (campaign_summary — campaign×date, CTR/CVR/CPA)  ──► Athena/QuickSight

오케스트레이션: Airflow (DAG 3종, BashOperator + docker exec)
  ad_lakehouse_silver   raw→Silver MERGE   매시간
  ad_lakehouse_gold     Silver→Gold 집계    매시간 (silver 의존)
  유지보수              compaction·expire·orphan  주기적(정확 값 §4)
```

### 핵심 기술 의사결정 (대안 비교 → 선택 근거)

| # | 결정 | 대안 | 선택·근거 |
|---|---|---|---|
| 1 | **레이크하우스 도입 여부** | OLTP DB + BI 툴 | 550K~1M행/일·100배 성장, 30일 소급 재집계(멱등), 다중 grain(대시보드 집계+ML 원자) → OLAP 명백, 플랫폼 필요. 단 실시간 스트리밍은 요구 아님(신선도 1h) → 마이크로배치로 시작(과잉설계 회피) |
| 2 | **테이블 포맷 = Iceberg** | Delta · Hudi · 전통 레이크(Hive+S3) | 광고 이벤트는 자주 바뀌고 파티션 전략도 운영 중 변경 → **파티션 에볼루션은 Iceberg만**. 엔진 호환성(Athena/Spark/Trino), AWS 생태계, 벤더락인 회피. 전통 레이크는 원자성 부재(중간 select가 깨진 상태 관찰)·읽기/리스트 비용 |
| 3 | **Kafka를 버퍼로** | 앱서버 → S3 직접 쓰기 | 연결 폭발(S3 초당 3500건 제한, 초과 503), 이벤트 유실(앱서버 OOM 시 버퍼 소실), 스몰파일 폭증, **장애 전파**(S3 지연이 광고 서빙 스레드 블로킹). Kafka **디커플링**으로 파이프라인·서비스 독립 운영 |
| 4 | **Bronze 파티션 = 수집시각** | 이벤트시각 파티션 | 지연 도착을 재파티션 없이 흡수, 쓰기 최적·원본 보존. Silver/Gold는 분석 편의로 이벤트시각 |
| 5 | **event_id 멱등 MERGE (2-pass)** | 단일 MERGE / append-only | at-least-once 중복·지연 도착을 같은 행에 수렴, 재처리 멱등. INSERT/UPDATE task 분리로 관측성↑ |
| 6 | **Silver = MOR** | CoW | 잦은 소량 업데이트의 write amplification 회피(쓰기 지연↓). 대가로 compaction/delete-rewrite 운영 필요 → daily_maintenance |
| 7 | **파일 포맷 = Parquet** | CSV/JSON | 컬럼 기반(필요 컬럼만 읽음), 스캔 바이트 ~1/10(Athena는 스캔량 과금), 압축·타입 안전. (유실 우려 시 Bronze는 JSON으로도) |
| 8 | **매니지먼트 = 별도 Airflow task** | 테이블 프로퍼티에 내장 | 프로퍼티만 넣으면 동작 여부 확인 어려움 → 별도 DAG task가 운영 가시성↑, 실패 시 원인 즉시 확인 |
| 9 | **오케스트레이션 = BashOperator + docker exec** | SparkSubmitOperator | 후자는 Airflow 이미지에 Spark 바이너리 필요해 비대 → 기존 spark-iceberg 컨테이너에 위임(랩 환경 단순) |

> **설계 판단 프레임(설계고민 문서 핵심)**: 도메인 → 요구사항 → (가능케 하는) 대안 → 우선순위(필수/선택)에 따라 **SLA·성능·비용**으로 최종 대안 선택. "도메인의 프로세스가 데이터 흐름과 연결되고, 돈과 직결되는 이벤트(전환)에 가용성·신뢰성을 더 배치한다"는 관점.

### 트러블슈팅 상세 (재현 가능 수준)
- **동시 쓰기 충돌 회피**: Iceberg snapshot isolation(각 writer가 base snapshot 기반 새 snapshot 원자 생성) + `max_active_runs=1`(DAG run 중첩 차단) + retries + **시간대 분리**(적재는 낮/매시, compaction·expire는 새벽). "ACID만 믿으면 안 되고 시간 분리까지 합쳐야 충돌 0."
- **롤백 절차(확인→검증→실행)**: BI 대시보드 이상 발견 → 지표 쿼리로 데이터 자체 문제인지(BI 캐시/리포트 버그 아닌지) 확인 → 타임트래블/SQL로 검증 → `rollback_to_snapshot`. "롤백이 잦으면 설계가 잘못된 신호", 보통 fix-forward 우선. 캐시 무효 절차·다운스트림 영향 인지 필요.
- **스냅샷 만료·고아파일**: 스냅샷 누적 = 메타데이터/스토리지 비용 → `expire_snapshots`(older_than 7일, retain_last N) + `remove_orphan_files`. expire는 커밋 작업, orphan 정리는 비동기. "expire/orphan 잡은 Spark 상시 리소스 대신 항시 엔진에 두기도" 등 운영 트레이드오프 학습.

### 불확실/검증필요 항목
1. **compaction target 파일 크기 — 테이블별로 다름**: raw/processed는 ≈128MB(`134217728`, min 50MB), 집계(campaign_summary)는 슬라이드에 512MB·57MB 등도 등장. about-me는 128MB로 기재. 이미지형 슬라이드 판독이라 오독 여지 있음 → **본인 repo 실제 코드값으로 확정 필요**. (이력서엔 확정 전 "테이블 특성에 맞게 적정 크기로 병합" 정도가 안전)
2. **BI 툴**: 강의 구현 = **QuickSight**(+Athena, DirectQuery). about-me §9 = "Trino/Athena". DuckLake는 미래 방향으로만 언급. 본인 repo에서 실제 쓴 것 확인 필요.
3. **DAG 스케줄·주기 정확값**: 적재(raw→silver→gold)는 매시간으로 일치. 유지보수 주기는 슬라이드 판독이 엇갈림(매일 03:00 vs compaction·expire 월별 / orphan 주별). **본인 repo DAG 파일로 최종 확인** 후에만 구체 cron 명시할 것.
4. **커밋 수**: about-me 43개 / GitHub 표기 44개(진행 중이라 변동).
5. **실측 성능·비용 수치 전무**: 쿼리 지연·스캔 절감·compaction 전후 파일 수 등은 랩 규모라 미측정. 포폴 강화하려면 본인 repo에서 실제 실행 후 스냅샷 summary/파일 수/실행시간을 **직접 측정해 캡처**하는 것이 가장 효과적(노션 "핵심 리스트"도 이 방향 강조).
