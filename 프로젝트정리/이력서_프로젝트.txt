[ 데이터 ]  1. 광고 데이터 레이크하우스  ( 개인 프로젝트 / 2026.05 ~ 2026.06 )
: 실시간 광고 이벤트(노출·클릭·전환)를 성능·비용 최적으로 적재·정제·집계하는 Medallion(Bronze/Silver/Gold) 데이터 레이크하우스
========================================

기술 스택 : Python·PySpark / Spark(Structured Streaming+배치)·Kafka·Airflow / Apache Iceberg(v2)·AWS Glue·Athena·QuickSight / S3·Docker Compose
설계 기준 : Criteo 공개데이터 30일 약 16.4M행(약 550K행/일) 기준

── 설계 결정 ───────────────────────────────────
■ 지연 도착 처리 (데이터 파이프라인)
설계 : Bronze를 수집시각 파티션으로, event_id 기준 멱등 MERGE INTO 적용
의도 : 전환이 클릭 후 최대 30일 늦게 도착(P95 3일) → 과거 파티션 재갱신 없이 수렴
결과 : 재처리 중복 0 보장, 쓰기·재처리 비용 절감

■ 쓰기 최적화 (스토리지)
설계 : MOR(Merge-on-Read) + MERGE를 insert_new_events·update_late_arrivals 2 task로 분리
의도 : 잦은 소급 갱신의 쓰기 지연·write amplification 최소화
결과 : 쓰기 지연 최적화, INSERT/UPDATE 분기 분리 관측

■ 테이블 운영 (운영 자동화)
설계 : rewrite_data_files 병합(약 128MB)·z-order·증분 MERGE 5일 윈도우, compaction·snapshot 만료·orphan 정리를 유지보수 DAG로 분리
의도 : 스트리밍 적재로 누적되는 스몰파일 → 읽기 성능 저하 방지
결과 : 읽기 최적화·운영 자동화

[ 데이터 AI ]  2. 지게차 충돌 위험 감지 "충돌주의보"  ( 팀 5명 / 2026.04 ~ 2026.05 )   ★ 최우수상
: 현장 CCTV 영상을 실시간 분석해 지게차-보행자 충돌을 2~3초 전에 예측·경고하는 산업안전 플랫폼
========================================

역할 : 팀장 + 검출·추적·위험예측 담당 (본인 커밋 55, 전체 2위)
기술 스택 : Python / YOLO11·BoT-SORT·Kalman(filterpy)·FastAPI / PostgreSQL / Docker Compose·Jenkins·EC2(GPU)·React·TS·WebSocket
설계 기준 : 실시간 영상 스트림(RTSP), FrameSlot 단일 슬롯·프레임 스킵으로 추론 부하 관리 (FPS·mAP는 [발표])

── 문제 해결 ───────────────────────────────────
■ 검출·추적 오탐 제거 (검출/추적)
문제 : 박스·트럭을 지게차로 오탐, 근접 두 객체를 한 박스로 묶음, 회전 시 ID switch
해결 : YOLO11n 2클래스 파인튜닝 + NMS·conf 임계 조정 + BoT-SORT match_thresh·ReID 튜닝
결과 : 오탐·박스 뭉침·ID switch 개선 (Before/After)

■ 궤적 끊김 보정 (추적)
문제 : 검출 실패(Lost) 구간에서 궤적이 끊겨 위험 예측 공백 발생
해결 : 등속(CV)·CTRV 칼만 외삽 적용, 1·2초 reachable zone 시각화
결과 : 끊김 없는 위험 예측

── 설계 결정 ───────────────────────────────────
■ 위험도 산정(RiskScore) (위험도)
설계 : CPA 기반 2D TTC + 거리·상대접근속도 결합 RiskScore, alert threshold 6단계 A/B 후 0.6 채택
의도 : TTC 단독은 정지·측면통과·근접 저속에서 오판 → 다신호 결합으로 보완
결과 : 조기경보와 오경보의 균형점 확보

[ 데이터 AI · 인프라 ]  3. 스마트팜 양액펌프 막힘 예지보전 "막힘주의보"  ( 팀 5명 / 2026.04.07 ~ 2026.04.23 )
: 양액 공급 설비의 막힘·열화 전조를 센서 시계열로 고장 전에 감지하고 원인(RCA)까지 짚는 예지보전 시스템
========================================

역할 : 백엔드·인프라 + 데이터·모델 (본인 커밋 33)
기술 스택 : Python / FastAPI·WebSocket·MQTT(Mosquitto)·TensorFlow Keras(AutoEncoder)·SHAP·Optuna / PostgreSQL·MinIO(S3) / Docker Compose(7컨테이너)
설계 기준 : 45개 변수 × 1분 단위 × 3개월 합성 시계열 / MQTT 실시간 + S3 Sink 10분 배치 이중 경로

── 문제 해결 ───────────────────────────────────
■ 학습 데이터 부재 돌파 (데이터)
문제 : 공개 고장 데이터가 없어 지도학습 자체가 불가
해결 : 도메인 조사 후 물리식 기반 45변수·3개월 합성 학습셋 직접 설계·생성
결과 : 문제 성립 + 막힘 에피소드 12/12 검출·평균 lead-time 약 28h  [측정]

── 설계 결정 ───────────────────────────────────
■ 이상탐지 구조 (이상탐지)
설계 : 4-도메인 분리 AutoEncoder(TF Keras×4) + 복원오차 6시그마 3단계 알람(2σ/3σ/6σ) + 디바운싱
의도 : 단일 임계는 헛알람이 많음 → 도메인 분리·다단계로 오탐 통제
결과 : 전 도메인 F1 0.95+·오탐률(FAR) 1.8% (단일센서 3.1% 대비 개선)  [측정]

■ 원인 진단(RCA) (RCA)
설계 : SHAP 피처 셀렉션 자동화 + 2층 RCA(도메인 voting → 피처 Top-3)
의도 : "이상"만이 아니라 "어느 신호가 원인인지"까지 제시
결과 : 특정 막힘 1건에서 원인 feature가 이상점수의 82% 차지 정량 입증  [측정]

[ 인프라 ]  4. Log-Land  ( 팀 5명 / 2023.08 ~ 2023.09 )   ★ 우수상(노브레이크)
: 사진의 EXIF(시간·위치)를 자동 추출해 지도·타임라인으로 보여주는 클라우드 다이어리
========================================

역할 : 초기 BE/FE·AWS → 팀원 이탈로 CI/CD·GitOps 인수·담당
기술 스택 : React·Node.js(Express) / AWS EKS·ECR·S3·RDS(MySQL)·CloudFront·Route53·ACM·WAF·CloudWatch / GitHub Actions·Kustomize·ArgoCD·Terraform
설계 기준 : EKS 멀티 AZ·On-Demand+Spot 노드그룹 설계 (비용 문제로 실운영은 single AZ)

── 문제 해결 ───────────────────────────────────
■ 배포 체계 전환 (CI/CD)
문제 : kubectl 직접 배포의 자격증명 노출·수동 롤백 리스크
해결 : (팀원 이탈로 CI/CD 인수) GitHub Actions(test·build·ECR)→Kustomize→ArgoCD(pull sync·Slack) GitOps 전환
결과 : 선언형·Git 단일 진실원본 배포 체계 구축

■ 장애 사전 감지 (모니터링)
문제 : 업로드 폭주 시 Pod OOM을 사후에야 인지
해결 : CloudWatch→SNS→Lambda→Slack 알람 + 리소스 튜닝
결과 : 사후 대응 → 사전 감지 전환

■ 컨테이너 통신 오류 (인프라)
문제 : 로컬에선 되던 FE→BE 호출이 EKS 환경에서 실패
해결 : Service/Ingress 라우팅·파드 배치 조정, 개발/테스트 DB 분리
결과 : 컨테이너 환경 통신 정상화

의도 : 카테고리/브랜드 기준은 규칙이 너무 복잡 → 상점 단위로 단순화

의도 : 소수점·한도에서 금액 불일치 방지, 공정 배분

의도 : 비정형 사용자 스펙을 투자·지원금 지표로 환산
결과 : 미래소득 기반 지원금 산정

[ 개발 ]  6. 댕동(DaengDong)  ( 팀 6명 / 2024.12 ~ 2025.01 )   ★ 수상(신한DS)
: 반려견 동반 여행을 계획·기록하고 커뮤니티로 공유하는 웹 서비스
========================================

역할 : 커뮤니티(게시판) 도메인 백엔드 + 진행관리 참여
기술 스택 : Java 17·Spring MVC·MyBatis / MySQL / JSP·jQuery/Ajax·WebSocket / AWS S3·OAuth2(Kakao/Google)·EC2·Jenkins

── 문제 해결 ───────────────────────────────────
■ 조회 성능 최적화 (조회)
문제 : 게시글 상세에서 이미지(1:N)·좋아요 집계 반복 조회(N+1)
해결 : MyBatis resultMap collection + LEFT JOIN으로 한 쿼리 매핑 (댓글은 별도 유지 판단)
결과 : 반복 조회 제거

■ 직렬화 오류 해결 (직렬화)
문제 : Ajax 응답이 계속 500
해결 : Tomcat 로그에서 InvalidDefinitionException 확인 → jackson-datatype-jsr310(JavaTimeModule) 등록
결과 : LocalDateTime 직렬화 오류 해소

── 설계 결정 ───────────────────────────────────
■ 이미지 저장 구조 (스토리지)
설계 : 이미지 DB 저장 대신 AWS S3 업로드 + URL(메타데이터) 분리
의도 : DB 용량·조회 부담 회피, 정적 파일은 오브젝트 스토리지가 적합
결과 : DB 부하 감소

[ 데이터 ]  7. Park-IN(파크인)  ( 팀 프로젝트 / 2025.06 ~ 2025.10 )   ★ 우수상(한국오라클)
: 공공데이터로 수요-공급 불균형을 계량화해 서울시 파크골프장 신규 입지 3곳을 추천한 데이터 분석 프로젝트
========================================

역할 : 문제 정의 + 분석 방법론 설계 + 접근성/커버리지 분석 개발
기술 스택 : Python(pandas·GeoPandas·Shapely·pyproj) / LASSO·AHP·베이지안·MCLP / Geocoding·좌표계 변환
설계 기준 : 서울 25개구 / 공공데이터 10여 종 / MCLP 후보 공원 30개

── 문제 해결 ───────────────────────────────────
■ 데이터 공백 돌파 (데이터 수집)
문제 : 핵심 변수(협회 회원수·시설 수용인원)가 미공개거나 구자료
해결 : 협회·시설에 직접 전화·메일로 2025 최신치 수집
결과 : 공개데이터 공백을 1차 수집으로 메워 분석 신뢰도 확보

── 설계 결정 ───────────────────────────────────
■ 가중치 편향 제거 (통계 모델링)
설계 : LASSO(데이터 변수선택) + AHP(전문가 설문)를 베이지안 통합 (LASSO=사전분포, AHP=우도)
의도 : 데이터 기반 vs 전문가 감(感)의 편향을 상호 보완
결과 : 통계 타당성 + 현실성 겸비한 수요지수 산출식

■ 입지 추천 (공간 최적화)
설계 : GeoPandas 5km 버퍼-행정동 교차 커버리지 분석(본인) + MCLP로 30개 후보 중 3곳 도출
의도 : 불균형 지수만으론 안 보이는 실제 접근성 사각지대를 시각화·최적화
결과 : 소외지역 불균형 최대 22.8%·3개구 평균 약 13% 완화, 왕복 이동 약 26~30분 단축  [발표]
